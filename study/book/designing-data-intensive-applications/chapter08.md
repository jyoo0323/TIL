# Ch.08 The Trouble with Distributed Systems

- 이번 챕터의 주 내용은 분산 시스템에서 발생할 문제들과 우리가 신뢰할 수 있는 것과 아닌 것들을 알아본다 함.
  - 믿을 수 없는 네트워크, 클록&타이밍 이슈 등을 어느 수준까지 회피 할 수 있는지

## Faults and Partial Failures
- Single Computer program은 보통 _전부 되거나, 아예 안되거나로 동작한다_
  - 하드웨어 적인 문제가 없다면 항상 같은 값을 출력한다 
    - 이러한 상황을 deterministic 하다고 한다
- 분산 시스템에선 예측 불가한 방식으로 죽은 시스템의 부분과 다른 멀쩡한 부분이 존재할 수 있다
  - 이런 상황을 부분 장애 (partial failure) 라고 하고 이런 문제들의 어려운 점은 nondeterministic 하다는 점이다.
### Cloud Computing and Supercomputing
- Supercomputing: 수퍼 컴퓨터 한대로 엄청난 연산을 하는 작업 (일기 예보 등)을 과학/공학적 연산 작업에 사용
- cloud computing: 명확한 정의는 없음. 다중 리전 데이터센터, 연결된 상용 컴퓨터, IP네트워크(ethernet으로 연결된), elastic/on-demand 리소스 할당, 사용량당(metered) 결제 등이 엮여있는 방식
  - 현대의 인터넷 서비스들의 대부분은 아래의 조건을 만족함:
    - 서비스 중단이 용납되지 않음 
      - 오프라인(배치성) 작업은 그럴 필요없음 -> 수퍼컴퓨터에서 하는 기후 시뮬레이션 작업 등
    - 연결된 상용 머신들로 구성되어 더 경재적으로 수퍼컴퓨터와 비슷한 퍼포먼스를 낼 수 있다는 장점이 있으나 더 높은 실패율을 가짐
- 현대의 분산 시스템은 의심과 비관주의 편집증적 사고가 도움이 크게 된다.

- **Building a Reliable System from Unreliable Components**
  - unreliable한 부분이 있다면 최대한 딱 그 정도만큼만 신뢰성이 있는 시스템이 될 것 같지만, 그렇지 않다 -> 믿음직스럽지 못한 컴포넌트들로 더 신뢰성있는 시스템 구축 가능!
    - ex: IP는 딜레이되고 유실되고 중복되거나 순서가 뒤바뀌는 경우도 있으나, 이 IP위에 만든 TCP는 신뢰성이 훨씬 높다. 유실된 패킷을 재전송, 중복을 제거해 패킷을 보낸 순서에 맞춰 재조립되도록 보장
  - 그러나 그래도 한계는 있다! TCP의 경우 네트워크 지연까지 해결은 불가

## Unreliable Networks
- 분산 시스템에선 네트워크만이 머신들간의 통신 창구이다.
  - 데이터 센터간의 통신에는 비동기 페킷 네트워크 (asynchronous packet network) 이며, 이는 송신 노드 쪽에서 수신하는 노드의 수신 여부를 보장 할 수 없는 환경이다
    - 이를 주로 핸들링하는 방법은 타임아웃 -> 그러나 타임아웃도 원인 파악은 불가. 그냥 기다림을 포기하는 것 뿐 
### Network Faults in Practice
- 네트워크 장비들을 중복으로 추가를 함은 결함을 줄여주지 못함 -> **네트워크 중단의 주 원인은 휴먼 에러이기 때문**
- 네트워크 장애는 드물더라도 발생 **가능**하다는 것이 중요
  - 네트워크 장애의 에러 처리가 정의되지 않으면 클러스터가 데드락에 빠지거나 데이터를 삭제할 수도 있음
  - 의도적으로 네트워크 문제를 만들어 시스템의 반응을 테스트해야 함 (Chaos Monkey 등)

### Detecting Faults
- 네트워크의 불확실성 때문에 노드가 동작 중인지 판단하기 어려움
- 일부 경우 피드백을 받을 수 있음:
  - 프로세스가 죽었지만 OS가 동작 중이면 TCP RST/FIN 패킷으로 연결 거부
  - 노드 프로세스가 죽으면 OS 스크립트가 다른 노드에 알림 (HBase가 사용하는 방식)
  - 네트워크 스위치 관리 인터페이스에 쿼리하여 하드웨어 수준 링크 장애 감지
  - 라우터가 IP 주소 도달 불가능 시 ICMP Destination Unreachable 패킷 전송
- 하지만 이런 피드백도 100% 신뢰할 수 없음 -> 결국은 타임아웃에 의존하게 됨

### Timeouts and Unbounded Delays
- 타임아웃을 얼마로 설정해야 하는가?
  - 긴 경우: 장애 감지가 오래 걸림, 사용자가 오래 기다려야 함
  - 짧은 경우: 일시적 슬로우다운을 장애로 오판할 위험이 높음
- 노드를 너무 일찍 죽었다고 선언하면 문제가 됨:
  - 실제로는 살아있는 노드가 작업 중이었는데 다른 노드가 이를 대체하면 같은 작업이 두 번 수행될 수 있음
  - 노드가 죽었다고 선언하면 해당 노드의 책임이 다른 노드로 전가 -> 이미 과부하인 시스템을 더 악화시킬 수 있음
  - 극단적인 경우 연쇄 장애(cascading failure): 모든 노드가 서로를 죽은 것으로 선언
- 가상의 시스템에서 최대 전달 지연 *d*, 처리 시간 *r*이 보장된다면 적절한 타임아웃은 *2d + r*
  - 하지만 대부분의 시스템은 이런 보장을 제공하지 않음: **비동기 네트워크는 무한 지연(unbounded delays)** 을 가짐

#### Network congestion and queueing
- 네트워크 패킷 지연의 변동성은 대부분 대기열(queueing) 때문:
  - 네트워크 스위치 큐: 여러 노드가 동시에 같은 목적지로 패킷을 보내면 스위치 큐에 대기, 큐가 가득 차면 패킷 드롭
  - OS 큐: CPU 코어가 모두 바쁘면 들어오는 요청은 OS에 의해 큐잉됨
  - VM 환경: VM이 CPU 코어를 양보하는 동안 네트워크 데이터가 버퍼링됨
  - TCP 흐름 제어(flow control): 수신 노드 과부하 방지를 위해 송신 속도를 제한 -> 송신 측 큐잉 발생
  - TCP 재전송: 타임아웃 내 ACK를 받지 못하면 패킷이 유실된 것으로 간주하고 자동 재전송 -> 지연 발생
- 실험적으로 타임아웃을 결정하는 것이 좋다
  - 네트워크 왕복 시간의 분포를 측정하고, 애플리케이션 특성을 고려해 장애 감지 지연과 조기 오판 위험 사이의 적절한 트레이드오프를 결정
  - 고정 타임아웃 대신 적응형 타임아웃을 사용하는 것이 더 좋음
    - 응답 시간과 지터(jitter)를 지속적으로 측정하여 자동으로 타임아웃 조정
    - Phi Accrual failure detector: Akka, Cassandra에서 사용
    - TCP 재전송 타임아웃도 유사하게 동작
### Synchronous Versus Asynchronous Networks
- 네트워크가 고정된 최대 지연으로 패킷을 전달하고, 패킷을 드롭하지 않는다면 분산 시스템은 훨씬 단순해질 것
  - 왜 하드웨어 수준에서 이를 해결하지 않는가?
    - 전화 네트워크(회선 교환, circuit-switched)** 와 비교:
      - 통화 시 고정된 대역폭의 **회선(circuit)** 이 할당됨, 큐잉 없음, 최대 지연이 고정됨 (**bounded delay**)
      - ISDN: 초당 4,000 프레임, 각 프레임에 16비트씩 할당 -> 250마이크로초마다 정확히 16비트의 오디오 전송 보장
    - TCP 는 왜 회선 교환 (circuit)을 쓰지 않는가?
      - 회선: 고정된 비트 레이트를 계속 잡고 사용함. (아무 요청이 오가지 않아도 해당 대역폭만큼이 사용됨)
      - 인터넷과 데이터센터 네트워크는 **bursty traffic** 에 최적화됨
        - 파일 전송, 웹 요청 등은 대역폭 요구가 불규칙 -> 회선은 비효율적 (대역폭 낭비)
        - TCP는 가용 네트워크 용량에 맞춰 동적으로 전송 속도를 조절
#### Latency and Resource Utilization
- 가변 지연은 동적 리소스 파티셔닝의 결과
  - 회선 교환: 리소스를 정적으로 분할 (10,000개 회선 중 하나를 독점, 나머지 9,999개가 비어있어도 동일한 대역폭)
  - 패킷 교환: 리소스를 동적으로 공유 -> 큐잉의 단점이 있지만 활용률을 극대화
- CPU의 경우: 스레드 간 동적으로 CPU 코어 공유 -> 활용률 극대화, 하지만 가변 지연
- 지연 보장은 가능하지만 비용이 높음 
  - 전용 하드웨어, 독점 대역폭 할당 등으로 가능
- 네트워크의 가변 지연은 자연 법칙이 아니라 **비용/편익 트레이드오프의 결과**

## Unreliable Clocks
- 시계(clock)와 시간은 분산 시스템에서 중요하지만 까다로운 주제
- 시간이 사용되는 용도:
  - Duration(기간) 측정: 요청 타임아웃 여부, 응답 시간의 99 백분위수, QPS 등
  - Point in time(시점) 측정: 기사 게시 시각, 캐시 만료 시점, 로그 타임스탬프 등
- 분산 시스템에서 시간이 까다로운 이유:
  - 통신이 즉각적이지 않음 -> 메시지 도착 시간은 항상 전송 시간보다 늦지만, 정확히 얼마나 늦었는지 알 수 없음
  - 각 머신마다 고유한 시계(쿼츠 크리스탈 오실레이터)가 있으며, 완벽히 정확하지 않음 -> 각 머신마다 약간씩 다른 시간
  - NTP(Network Time Protocol)로 동기화 가능하지만 완벽하지 않음

### Monotonic Versus Time-of-Day Clocks

#### Time-of-day clocks
- 직관적으로 기대하는 시계: 현재 날짜와 시간을 반환 (wall-clock time)
- Linux: `clock_gettime(CLOCK_REALTIME)`, Java: `System.currentTimeMillis()`
- epoch(이팍) 1970-01-01 00:00:00 UTC 이후 경과한 초/밀리초를 반환
- NTP로 동기화 -> 다른 머신의 타임스탬프와 (이상적으로는) 동일한 의미
- 문제점:
  - NTP 서버와 너무 차이가 나면 강제 리셋될 수 있음 -> 시간이 뒤로 갈 수 있음
  - 윤초(leap second)를 무시하는 경우가 많음
  - 경과 시간 측정에 부적합

#### Monotonic clocks
- 기간(duration)을 측정하는 데 적합: `clock_gettime(CLOCK_MONOTONIC)` (Linux), `System.nanoTime()` (Java)
- 항상 앞으로만 감 (time-of-day clock은 뒤로 갈 수 있음)
- 절대값은 무의미 (컴퓨터 시작 이후 경과 나노초 등)
- **두 다른 컴퓨터의 단조 시계 값을 비교하는 것은 의미 없음**
  - NTP는 단조 시계의 진행 속도(frequency)를 조정할 수 있지만 (slewing), 점프시키지는 않음
  - 분산 시스템에서 경과 시간 측정 (타임아웃 등)에 적합: 노드 간 동기화가 불필요하고 약간의 측정 부정확성에 민감하지 않음
  
### Clock Synchronization and Accuracy
- 단조 시계는 동기화가 필요 없지만, time-of-day 시계는 NTP 등 외부 시간 소스와 동기화가 필요
- 시계가 정확하지 않을 수 있는 이유들:
  - 쿼츠 시계의 드리프트(drift): 온도에 따라 빨라지거나 느려짐. Google은 200 ppm 드리프트를 가정 -> 30초마다 재동기화해도 6ms 드리프트
  - NTP 서버와 차이가 너무 크면 동기화 거부 또는 강제 리셋
  - 방화벽이 NTP 트래픽을 차단하면 눈치채지 못한 채 드리프트가 누적됨
  - NTP 동기화 정확도는 네트워크 지연에 제한됨 -> 인터넷에서 최소 35ms 오차, 네트워크 지연 스파이크 시 수초
  - 일부 NTP 서버가 잘못 설정되어 수시간 차이의 시간을 보고할 수도 있음
  - **윤초(leap second)**: 1분이 59초 또는 61초가 되는 경우 -> 윤초를 고려하지 않은 시스템에서 장애 유발
  - VM 환경: 하드웨어 시계가 가상화되어 VM이 일시 정지되면 시계가 갑자기 점프
  - 사용자 기기의 시계는 의도적으로 잘못 설정될 수 있음 (게임 타이밍 우회 등)
- 높은 정확도를 위해서는 GPS 수신기, PTP(Precision Time Protocol) 등 사용 가능은 하나 상당한 리소스가 필요함

### Relying on Synchronized Clocks
- 시계는 단순하고 사용하기 쉬워 보이지만, 많은 함정이 존재
  - ex: 하루가 정확히 86,400초가 아닐 수 있고, time-of-day 시계가 뒤로 갈 수 있으며, 노드마다 시간이 다를 수 있음
- 부정확한 시계는 **감지하기 어려움**: 
  - 결함이 있는 NTP 클라이언트는 겉으로는 잘 작동하는 것처럼 보이면서 시간만 점점 틀어짐 -> 조용한 데이터 손실 가능
- **시계 오프셋을 노드 간에 모니터링하여, 너무 많이 벗어난 노드는 클러스터에서 제거해야 함**

#### Timestamps for ordering events
- 분산 데이터베이스에서 두 클라이언트가 쓰기를 하면, 어느 것이 먼저인지를 시계로 판단하는 것은 위험
- LWW (Last Write Wins) 충돌 해결 전략의 문제:
  - 시계 스큐(clock skew)로 인해 인과적으로 나중인 쓰기가 더 이른 타임스탬프를 가질 수 있음
  - 데이터가 조용히 사라질 수 있음 (빠른 시계를 가진 노드의 값이 느린 시계 노드의 값을 덮어씀)
  - 순차적으로 발생한 쓰기와 진정으로 동시에 발생한 쓰기를 구분할 수 없음
  - 같은 밀리초 내 두 노드가 독립적으로 같은 타임스탬프를 생성할 수 있음
- 이벤트 순서를 정하기 위해 "가장 최근" 값을 유지하는 것은 매력적이지만, "최근"의 정의가 로컬 time-of-day clock에 의존하므로 잘못될 수 있다
- **논리적 시계(logical clocks)**: 카운터를 증가시키는 방식으로, time-of-day나 경과 시간을 측정하지 않고 이벤트의 상대적 순서만을 추적해 더 안전한 대안
  - 이에 비해 time-of-day, monotonic clock은 **물리적 시계(physical clocks)** 라 함

#### Clock readings have a confidence interval
- 시계 값을 시점(point in time)이 아닌 **신뢰 구간(confidence interval)** 내의 범위로 생각하는 것이 적절
  - 예: "현재 시간은 10.3~10.5초 사이일 확률이 95%"
  - 쿼츠 드리프트, NTP 서버와의 마지막 동기화 시간, 네트워크 왕복 시간 등에 따라 불확실성이 결정됨
- 대부분의 시스템은 이 불확실성을 노출하지 않음: `clock_gettime()` 호출 시 오차 범위를 알려주지 않음
  - 불확실성을 명시적으로 노출시키는 Google Spanner의 TrueTime API
    - `[earliest, latest]` 두 값을 반환 -> 실제 현재 시간이 이 구간 내에 있음을 보장

#### Synchronized clocks for global snapshots
- [Snapshot Isolation](chapter07.md)은 작고 빠른 읽기-쓰기 트랜잭션과 크고 오래 걸리는 읽기 전용 트랜잭션을 모두 지원하는 매우 유용한 기능
  - 단조 증가하는 트랜잭션 ID가 필요함
  - 단일 노드: 간단한 카운터로 가능
  - 분산 환경: 전역적으로 단조 증가하는 트랜잭션 ID 생성이 어려움 (조율이 필요)
    - Snowflake 같은 분산 시퀀스 생성기는 근사적으로만 단조 증가 -> 인과성 보장이 안 될 수 있음
- **Spanner의 접근법**: 동기화된 time-of-day 시계의 타임스탬프를 트랜잭션 ID로 사용
  - TrueTime의 신뢰 구간을 활용: 두 트랜잭션의 신뢰 구간이 겹치지 않으면 순서가 확실
  - 신뢰 구간이 겹칠 때만 순서가 불확실 -> **의도적으로 신뢰 구간의 길이만큼 대기 후 커밋**
    - 대기 시간을 줄이기 위해 Google은 각 데이터센터에 GPS 수신기 또는 원자시계를 배치 -> 시계 불확실성을 7ms 이내로 유지
- Spanner 외에 이 아이디어를 구현한 주류 데이터베이스는 아직 없음

### Process Pauses
- 분산 시스템에서 위험한 시계 사용의 또 다른 예: **리스(lease)** 기반 리더십
  - 단일 리더 데이터베이스에서 리더는 다른 노드로부터 리스를 얻어 리더십을 유지
  - 리스가 만료되기 전에 갱신해야 함
- 리스 기반 코드의 위험: 스레드가 리스 만료 확인과 요청 처리 사이에 예상치 못한 일시 정지(pause) 가 발생할 수 있음
  - 일시 정지 동안 리스가 만료되고 다른 노드가 리더가 될 수 있지만, 원래 노드는 이를 모름
- 스레드가 오랫동안 일시 정지될 수 있는 이유들:
  - GC(Garbage Collector): "stop-the-world" GC pause는 수분간 지속될 수도 있음. "concurrent" GC도 완전히 병렬은 아님
  - 동기식 디스크 I/O: 디스크 접근으로 스레드가 블로킹. Java의 클래스 로더 등 예기치 않은 곳에서 디스크 접근 발생 가능. 네트워크 파일시스템(NFS, EBS)이면 네트워크 지연까지 영향
- 이 모든 것이 스레드를 임의의 시점에 선점(preempt) 하고 나중에 재개할 수 있음
  - 단일 머신의 스레드 안전성과 유사한 문제, 하지만 분산 시스템에는 공유 메모리가 없고 비신뢰적 네트워크를 통한 메시지만 존재
- 노드는 언제든 임의의 시간 동안 일시 정지될 수 있다고 가정해야 함

#### Response time guarantees
- hard real-time 시스템: 데드라인 내 응답을 보장
  - 항공기 제어, 로켓, 로봇, 자동차 등 안전이 중요한 시스템
  - RTOS(Real-Time Operating System) 필요, 라이브러리 함수의 최악 실행 시간 문서화, 동적 메모리 할당 제한 등
  - 매우 비싸고, **대부분의 서버 사이드 데이터 처리에는 적합하지 않음**
- 대부분의 프로그래밍 언어와 OS에서 스레드와 프로세스는 **무한정 일시 정지될 수 있음**

#### Limiting the impact of garbage collection
- GC 영향을 줄이는 실용적 접근법:
  - GC pause를 **계획된 짧은 장애(planned brief outage)** 로 취급: 
    - GC가 임박하면 해당 노드로의 요청을 중단하고 다른 노드에서 처리, GC 후 재개
  - 또는 수명이 짧은 객체만 GC 하고 (빠르게 수집 가능), 수명이 긴 객체는 데이터가 충분히 쌓이면 프로세스를 재시작하여 처리

## Knowledge, Truth, and Lies
- 분산 시스템에서 노드는 자기 자신의 상태조차 확신할 수 없음
  - 네트워크를 통해 메시지를 주고받는 것만으로 다른 노드의 상태를 파악해야 함
  - 개별 노드의 판단은 신뢰할 수 없음 -> **분산 시스템에서 진실은 다수결(quorum)로 정의됨**

### The Truth Is Defined by the Majority
- 분산 시스템에서는 단일 노드를 신뢰할 수 없으므로, **다수의 노드가 동의하는 것을 진실로 취급**
  - 과반수(quorum): 보통 전체 노드의 절반 이상이 동의하면 결정이 내려짐
- 노드가 자신이 살아있다고 믿어도, **나머지 과반수가 죽은 것으로 판단하면 그 노드는 죽은 것으로 취급됨**

#### The leader and the lock
- 자주 등장하는 상황: 하나의 노드만 특정 역할을 해야 하는 경우
  - 데이터베이스 파티션의 리더 (split brain 방지)
  - 특정 리소스/객체에 대한 락을 보유한 클라이언트만 쓰기 허용
  - 특정 사용자 이름을 유일하게 등록한 사용자만 사용 가능
- 분산 시스템에서 이 제약을 구현하려면 주의가 필요:
  - 노드가 자신이 "선택된 자"라고 믿더라도 과반수가 동의하지 않을 수 있음
  - 리스를 가진 노드가 일시 정지 -> 리스 만료 후 다른 노드가 리더가 됨 -> 원래 노드가 재개되어 아직 리더라 착각하고 안전하지 않은 작업 수행

#### Fencing tokens
- 위의 문제를 해결하기 위한 방식이 이 fencing token이다.
- 리스/락 서버가 락을 부여할 때마다 **단조 증가하는 펜싱 토큰(fencing token)** 을 함께 반환
  - 클라이언트가 쓰기 요청을 보낼 때 자신의 펜싱 토큰을 포함시킴
  - 리소스(저장 서비스)가 이전에 처리한 토큰보다 낮은 토큰을 가진 요청은 거부
- ex: 클라이언트1이 토큰 33으로 락을 얻고, 일시 정지 중에 락이 만료됨. 클라이언트2가 토큰 34로 락을 얻고 쓰기 수행. 클라이언트1이 재개되어 토큰 33으로 쓰기를 시도하면 저장 서비스가 이미 토큰 34를 처리했으므로 거부됨
- **서버 측에서의 토큰 검증**: 클라이언트가 올바르게 동작한다고 가정하지 않음
- ZooKeeper는 `zxid`나 `cversion`을 펜싱 토큰으로 사용할 수 있음

### Byzantine Faults
- 펜싱 토큰은 의도치 않은 오류(일시 정지 등)를 방지할 수 있지만, 노드가 의도적으로 *거짓 메시지*를 보내는 경우는 막을 수 없음
  - 거짓 메세지란?
    - 노드가 프로토콜의 규칙을 어기고 임의의(arbitrary) 응답을 보내는 것입니다. 예를 들어:
    - 실제로 메시지를 받지 않았는데 "받았다"고 주장하는 경우
    - 데이터를 저장하지 않았는데 "저장했다"고 응답하는 경우
    - 가짜 fencing token을 만들어 시스템의 보호 장치를 우회하는 경우
    - 잘못된/조작된 데이터를 다른 노드에게 전파하는 경우
- **비잔틴 장애(Byzantine fault)**: 노드가 프로토콜을 따르지 않거나 거짓 메시지를 보내는 경우 (악의적 또는 소프트웨어 버그로 인해)
  - 이런 환경에서도 올바르게 동작하는 시스템을 **비잔틴 내결함(Byzantine fault-tolerant)** 이라 함
  - **비잔틴 장군 문제(Byzantine Generals Problem)**: 신뢰할 수 없는 채널을 통해 합의에 도달하는 문제
- 대부분의 서버 사이드 시스템에서는 비잔틴 내결함성이 **불필요**:
  - 웹 애플리케이션: 서버가 모든 입력을 검증할 수 있음. "비잔틴"인 것은 클라이언트(사용자)이지 서버 노드가 아님
  - 조직 내부 시스템: 모든 노드가 같은 조직에 의해 운영되므로 신뢰 가능
- 비잔틴 내결함성이 필요한 경우:
  - 항공우주 -> 방사선에 의한 메모리 손상으로 예측 불가능한 응답 가능
  - 블록체인/P2P 네트워크 -> 참가자가 서로 신뢰하지 않는 환경
- 소프트웨어 버그가 비잔틴 장애와 유사하게 동작할 수 있으나, 보통 같은 소프트웨어를 실행하는 모든 노드에서 동일한 버그가 발생하므로 비잔틴 내결함성으로 해결되지 않음
- **약한 형태의 거짓말(lying) 방지**는 실용적:
  - 네트워크 패킷의 하드웨어 수준 손상 -> 체크섬으로 감지
  - 사용자 입력의 위생 처리(sanitization): SQL 인젝션, XSS 방지
  - NTP 클라이언트: 여러 서버를 쿼리하여 이상치(outlier) 제거

### System Model and Reality
- 알고리즘을 설계하고 증명하기 위해 시스템 모델(system model) 로 가정을 형식화:
- 타이밍 모델:
  - **동기 모델(Synchronous model)**: 네트워크 지연, 프로세스 일시 정지, 시계 오차에 상한이 있다고 가정 -> 비현실적
  - **부분 동기 모델(Partially synchronous model)**: 대부분의 시간은 동기적이나, 때때로 한계를 초과할 수 있음 -> **현실적인 가정**
  - **비동기 모델(Asynchronous model)**: 타이밍에 대한 어떤 가정도 하지 않음. 시계조차 없을 수 있음 (매우 제한적)
- 노드 장애 모델:
  - Crash-stop: 노드가 한번 크래시하면 영원히 돌아오지 않음
  - Crash-recovery: 노드가 크래시 후 다시 돌아올 수 있음. 메모리는 유실되지만 안정 저장소의 데이터는 보존됨 -> **_가장 현실적인 가정_**
  - Byzantine(arbitrary) faults: 노드가 어떤 것이든 할 수 있음 (거짓말 포함)
- 부분 동기(partially synchronous) + crash-recovery 가 가장 유용함

#### Correctness of an algorithm
- 알고리즘의 올바름은 **속성(properties)** 으로 정의:
  - 안전성(Safety): "나쁜 일이 일어나지 않는다" - 위반되면 특정 시점을 가리킬 수 있음. **항상 성립해야 함**
    - 예: 펜싱 토큰의 유일성, 단조 증가성
  - 활동성(Liveness): "좋은 일이 결국 일어난다" - 특정 시점을 가리킬 수 없음. 일시적으로 성립하지 않을 수 있으나 **결국은 성립해야 함**
    - 예: 가용성 - 요청이 결국은 응답을 받음
- 안전성과 활동성을 구분하는 것이 중요: 장애 상황에서도 안전성은 반드시 보존되어야 하지만, 활동성은 일시적으로 위반될 수 있음

#### Mapping system models to the real world
- 시스템 모델은 현실의 단순화이지만, 알고리즘의 올바름을 추론하는 데 매우 유용
- 실제 구현에서는 모델 너머의 엣지 케이스도 처리해야 함:
  - 예: crash-recovery 모델은 안정 저장소의 데이터가 보존된다고 가정하지만, 디스크 데이터가 손상될 수도 있음.. 이런 경우는 어카나?
  - 비잔틴이 아닌 모델이라도 소프트웨어 버그가 예상 보장을 위반할 수 있음
- 모델은 완벽하지 않지만, 알고리즘이 올바르게 동작할 수 있는 속성을 파악하는 데 매우 유용하며, 이를 통해 다양한 장애를 체계적으로 처리할 수 있는 인프라를 구축 가능
