### Data Structures That Power Your Database
- 이 장에서의 Log의 의미: 연속된 추가 전용 레코드 (애플리케이션 개발자 입장에서의 로그보다 더 일반적인 로그의 의미).
  - 읽기 쉬운 포맷일 필요없고, 특정 읽기 프로그램을 통해서만 읽히게 만들기도 함
- 인덱스 (색인): **데이터베이스에서 특정 키의 값을 효율적으로 찾기 위한 데이터 구조**
  - 이정표 역할을하는 부가적인 메타데이터를 유지해 데이터의 위치를 특정
  - 여러가지 방법으로 검색을 해야하면 여러 인덱스가 필요
  - 인덱스의 추가와 삭제는 쿼리 성능에 영향을 줌
    - 쓰기는 무조건 느려지고, 읽기는 어떻게 인덱스를 잡았느냐에 따라 영향이 다름 -> 키 트레이드 오프
- Hash Index:
  - key-value storage ~= dictionary -> 보통 해시 테이블로 구현함 
  - 모든 키를 인메모리 해시에 저장하고, 그 키에 해당하는 로그 파일의 주소를 값으로 저장해 사용하는 방식
  - 키가 자주 업데이트 되는 환경에 잘 어울리는 구조.
  - 저장하는 파일이 무한히 커지는 것을 방지하기 위해 파일을 세그먼트로 나누고, **컴팩션(compaction)**을 통해 로그에 중복된 키를 제거
  - Append only(추가전용) 로그의 장점:
    - append 와 segment merging 은 sequential write op 이기에 랜덤한 write 보단 대체적으로 빠르다
    - 동시성 관리와 크래시 리커버리:
      - 키의 값이 수정되다 크래시가 발생했을 떄 등을 고려할 필요가 없어짐
  - Hash Table 의 한계:
    - 해시 테이블이 메모리 보다 작아야함. 키가 굉장히 많아지면 쓸수가 없음 
    - 범위 질의에 약함 -> 범위로 질의할 때 결국 풀스캔을 해야함

#### Constructing && Maintaining SSTables
  - 들어오는 쓰기 요청은 순서 보장 없이 들어오는데 어케 순서를 지켜서 SSTable의 Sorting을 지키는지?
    - red-black / AVL tree 등 트리 자료 구조를 통해 구현 가능
      - memtable이라는 인 메모리 트리에 쓰기 요청의 키를 삽입 
        - 이 멤테이블이 너무 커지면 디스크에 저장하고, 그 저장된 파일이 SSTable의 가장 최신 세그먼트가 됨
      - 읽기 요청시 현재 멤테이블에 키가 있는 지 확인하고, 없으면 최신순으로 세그먼트들을 검색
      - 백그라운드에서 별도로 세그먼트의 병합과 컴팩션을 진행
  - 유일한 문제: 디비에 크래시가 나서 다운될 경우 인메모리에 있던 가장 최신의 멤테이블은 디스크에 작성되지 못해 사라져버림
    - 이를 해결하기 위해 모든 쓰기 요청에 대한 로그 파일을 만들어둠 (복구용)

#### LSM Tree (Log-Structured Merge-Tree)
  - “SSTable 파일들을 계층(레벨)로 관리하면서” 백그라운드 compaction(merge)을 통해 정리하는 구조
  - 위 Constructing && Maintaining SSTable 에서 말하는 인덱싱 구조를 LSM tree 라 칭함.
  - 재정리
    - write 작업의 흐름:
      - write 요청 → (in-memory 정렬 구조에 반영) → 일정량이면 SSTable로 flush → 컴팩션으로 여러 SSTable을 병합/정리(중복키 제거)
      - 이러한 쓰기 프로세스는 엄청나게 높은 write throughput 을 지원해줌
    - Read 작업의 흐름:
      - 최신(메모리/최신 SSTable)부터 찾아보고, 없으면 다음 레벨로 내려가며 확인
    - compaction에서 발생하는 작업:
      - 중복 키 제거(최신 값만 유지)
      - tombstone 반영(삭제 반영)
      - 파일 수/레벨 유지(읽기 비용 제어)
  - LSM tree의 최적화 필요 사항:
    - 존재하지 않는 키를 찾을 경우:
      - 멤테이블 -> 최신 세그먼트 -> ... -> 모든 세그먼트 탐색이 필요
      - 해결하기 위한 방법:
        - Bloom Filters: 대략적인 Set의 내용물들을 위한 메모리 효율적 자료 구조로, 데이터 베이스내에 해당 키의 존재 여부를 알려 줄 수 있다.
      - SSTable이 compact 되고 병합되는 전략들:
        - size-tiered compaction
          - 새로 생기고 더 작은 SSTable을 기존에 존재하는 더 크고 오래된 SSTable에 병합
        - level compaction
          - 키 범위는 더 작은 SSTable로 나누고 오래된 데이터를 새로운 레벨로 이동해 컴팩션을 점진적으로 진행해 디스크 공간을 덜 사용함

#### B-Tree (Balanced Tree)
- **가장 널리 사용되고 있는 인덱스 구조**
  - 거의 대부분의 관계형 데이터베이스에서 사용하는 색인 구조
- 4KB 정도의 고정 크기 불록/페이지로 나누고 한 번에 하나의 페이지에 읽기 또는 쓰기를 한다  
- branching factor: 하나의 페이지에서 하위 페이지를 참조하는 수
  - ex: ref-100-ref-200-ref 요런 페이지가 있다면 계수는 3이됨 (100<x, 100<=x<200, 200<=x 가 브랜치로 존재)
- Read 작업의 흐름:
  - 키를 통해 요청 -> 루트 (시작 페이지)의 범위를 찾아봄 -> 다음 페이지에서 범위 검색 -> ... -> leaf page를 전부 스캔해 최종 값을 선택
- write 작업의 흐름:
  - 업데이트: 기존 참조를 전부 유지한 체 키가 포함된 리프 페이지를 찾아 값을 변경 다시 디스크에 쓰기
  - 삽입: 해당 키 범위의 페이지를 찾아 추가. 이 때 만약 공간이 부족하면 페이지를 두개로 분할해 부모 페이지를 업데이트
- Making B-Tree Reliable:
  - B-Tree에서는 기존 페이지를 overwrite in-place 하는데, 이는 LSM의 세그먼트 추가와는 대조적
  - WAL(Write Ahead Log), AKA redo-log: 
    - LSM Tree 때와 마찬가지로, 시스템의 장애 복구를 위한 데이터 구조가 필요함
    - 트리 페이지에 변경된 내용을 적용하기 전 모든 B 트리의 변경사항을 기록하는 append only file로 복구될 때 일관성있는 상태로 B tree 를 복원하는데 사용됨
  - Latch: 경량 잠금
    - 트리의 데이터 구조를 보호하기 위해 사용하는 락으로 동시성의 제어를 위해 사용됨.
- 최적화:
  - Copy-on-write scheme: WAL 대신 변경된 페이지를 새 위치에 쓰고 부모를 업데이트
  - 키 축약: 키를 축약해 경계만 남겨서 브랜칭 팩터를 높이고, 트리의 깊이를 줄이는 기법.

#### Comparing B-tree and LSM tree
- Write: LSM Tree > B Tree
- Read: LSM Tree < B-Tree
  - LSM tree는 컴팩션 단계에 있는 여러가지 데이터 구조와 SSTable 세그먼트들을 읽어야 하기 때문에 느림
- Advantages of LSM Tree
  - Write Amplification (쓰기 증폭): 한번의 DB write이 디스크에 여러 번 쓰이는 현상
  - B tree 는 WAL 에 한번, 페이지에 한번 Write 를 해야함
    - Write Amplification 이 높아짐
  - LSM 도 컴팩션&병합으로 반복적 쓰기가 존재하지만, 순차적으로 write되고 압축을 진행하기 때문에 상대적으로 쓰기 증폭이 낮음
- Downsides of LSM Tree
  - 컴팩션이 읽기/쓰기 성능에 영향을 줌
    - 전체적인 것은 아니고 상위 백분위 (p95 이상?) 의 요청이 상대적으로 예측이 어렵다
  - 높은 처리량이 요구되는 Write 작업을 해야할 때 컴팩션이 쓰기를 못 따라가는 현상이 존재
    - 미병합된 세그먼트가 증가 -> 디스크 부족 + 읽기 성능도 저하됨
  - 트랜잭션 시맨틱이 떨어짐
    - 여러 세그먼트에 같은 키의 다중 복사본이 존재할 수도 있기 때문

#### Other Indexing Structures
- Secondary Index (보조 색인)
  - PK외의 다른 열에 대한 인덱스
  - 키가 유일하지 않기에 값을 ID 목록으로 만들거나, 키에 row ID 를 붙여서 유일하게 만들 수 있음
  - B tree & LSM tree 모두 보조 인덱스로 사용이 가능함
- 인덱스 내부에 값 저장하기
  - 인덱스의 키는 검색하는 대상
  - 인덱스의 값은 실제 row 혹은 해당 row를 가리키는 참조
  - Heap File: 인덱스의 값은 실제 데이터 위치의 참조 → 데이터는 별도 힙 파일에 저장 (중복 방지)
  - Clustered Index: 인덱스 안에 실제 행 데이터를 직접 저장. 예: MySQL InnoDB의 primary key 
  - Covering Index (인덱스에 포함된 열): 일부 열만 인덱스에 포함 → 해당 쿼리는 인덱스만으로 응답 가능 
  - 클러스터드/커버링 인덱스의 트레이드 오프
    - PRO: 읽기는 빨라짐 
    - CONS: 저장 공간 증가 + 쓰기 오버헤드 + 트랜잭션 일관성 유지 부담
- Multi-Column indices (다중 열 인덱스):
  - Concatenated Index: 여러 필드를 하나의 키로 합치는 인덱스
    - 첫번째 컬럼으로는 단독 인덱스로 사용이 가능하지만, 그 외 컬럼들은 단독으로는 검색 불가
  - Multi-dimensional index: 여러 열을 동시에 쿼리할 때 사용하는 인덱스
    - ex: 위도 경도 기반으로 레스토랑 탐색시
    - R-Tree 라는 공간 인덱스를 이용해 구현
    - RGB 조합을 통한 색상, 날짜-온도 조합 검색 등에서도 이러한 인덱스를 활용
- Full-Text search and Fuzzy indexes:
  - Full-Text search: 동의어 확장, 문법 변형 무시, 근접 단어 검색, 오타 허용 등을 포함한 검색 방법
    - Luccene 을 통해 가능하다고 함. 
- Keeping Everything in memory
  - Disk가 메모리에 비해 가지는 장점들:
    - Durability: 전원이 나가도 데이터가 날아가지 않음
    - RAM에 비해 기가바이트당 단가가 쌈
  - RAM의 단가가 싸지면서 데이터셋이 작은 경우에는 메모리에 전부 올리는게 가능해짐
- 인메모리 DB:
  - 분류
    - 캐시 전용 (Memcached) : 재시작 시 데이터의 손실을 허용
    - 내구성 지향 (Redis, VoltDB, Couchbase 등):
      - 디스크에 변경 로그 기록, 주기적으로 스냅샷, 복제를 남김으로 내구성을 확보
      - 읽기는 순수하게 항상 메모리에서 한다
  - 장점: 
    - 디스크 기반 인덱스에서는 구현하기 어려운 데이터 모델을 사용 가능
      - redis 의 경우 priority-queue, set 등
    - 디스크에 저장할 때 보다 빠른 성능을 가진다.
      - 다만 이는 메모리에서 꺼내기 때문만은 아니고 디스크에 저장하기 위한 형태로 부호화(encoding)하는 오버헤드가 없어서 그런 것 같다 
  - Anti-caching: 인메모리DB 아키텍처를 가용 메모리보다 큰 데이터 셋으로 확장하는 접근법
    - 가장 최근에 사용하지 않은 데이터를 메모리에서 디스크로 보내고 나중에 다시 접근할 때 메모리에 적재하는 방식.
    - 전체 인덱스가 메모리에 있어야 사용이 가능함
  - NVM (non-volatile memory)이 널리 보급되면 스토리지 엔진 설계에 추가적인 변화가 필요할 수 있다.
    - 현재는 연구 단계임

### Transaction Processing or Analytics?
- OLTP (On-Line Transaction Processing)
  - 소량의 레코드를 키로 조회 인덱스 기반
  - 사용자 입력에 따른 삽입/수정
  - 대화식 -> 낮은 지연 시간이 필수
  - 데이터셋의 규모는 상대적으로 작다 (GB ~ TB)
  - 주로 최신 상태의 데이터를 다룸
  - 유저가 주 사용자
- OLAP (On-Line Analytics Processing)
  - 대량의 레코드를 스캔, 몇 개 열만 읽어 집계(COUNT, SUM, AVG)
  - ETL 또는 이벤트 스트림으로 대량 적재 
  - 비즈니스 분석가가 주 사용자 → 의사결정 지원 
  - 데이터셋의 규모가 큼 (TB ~ PB)
  - 시간에 걸쳐 발생한 이벤트의 이력
  - 비즈니스 인텔리전스를 제공해 더 나은 비즈니스 의사 결정에 도움을 줌

#### Data Warehousing
- OLTP에 영향을 주지 않고 마음껏 질의할 수 있는 개별 데이터베이스
  - 사내의 모든 다양한 OLTP 시스템에 있는 데이터의 읽기 전용 복사본
- OLTP 데이터 베이스를 그대로 가져오기 보단 분석 친화적 스키마로 변환하고 깨끗히 정리한 다음 데이터 웨어하우스에 적재됨
  - 이 과정을 ETL (Extract-Transform-Load) 라고 함
- OLTP 데이터베이스와 데이터 웨어하우스의 차이점:
  - 데이터 웨어하우스에서는 일반적인 관계형 모델을 사용한다고 함
  - 둘은 OLTP와 OLAP의 차이에 맞게 질의 패턴을 가져가기 때문에 서로 지원하는 영역이 다르다.
    - OLTP 디비 -> 트랜잭션 처리 위주
    - 데이터 웨어하우스 디비 -> 분석 작업 부하 위주
  - 최근에는 양쪽을 지원하기 보다 한쪽에 몰빵하는게 추세이고, sql-on-hadoop 오픈소스 프로젝트들이 성행하고 있다고 한다

#### Stars and Snowflakes: Schemas for Analytics
- 분석에서는 데이터 모델의 다양성이 적다.
- Star Schema (dimensional-modeling): 데이터 웨어하우스에서 사용하는 분석용 스키마 
  - fact table에 발생한 이벤트들을 모아서 저장한다
    - fact table의 각 row는 어떠한 행위가 일어났는지등에 대한 기록.
    - fact 테이블은 외래키로 수많은 차원 테이블들을 가지고 있다.
      - 차원테이블의 row는 누가, 언제, 어디서, 무엇을, 어떻게, 왜를 나타낸다
    - _분산 데이터 베이스 시스템에선 어떻게 설계하는걸까?_
  - 테이블의 관계를 시각화시 사실 테이블을 중심으로 차원 테이블로 둘러싸고 있기 때문에 이러한 이름이 붙었다
- Snowflake schema: 별모양 스키마의 변형으로 차원이 하위 차원으로 더 세분화된 버전
##### Column-Oriented Storage
- 사실 테이블의 컬럼이 너무 많아지면, 실제로 질의에 사용되는 컬럼은 4,5 개인데 존재하는 컬럼은 100개가 넘는다. 
  - 이러한 구조가 일반적인 OLTP에서 사용하는 row 중심적 스토리지
  - 그러나 질의시에는 필요없는 열까지 전부 디스크에서 읽어 메모리에 로드한다
    - 이러한 구조의 효율성을 높이기 위한 방식
- 해결 방식: 한 컬럼의 모든 값을 함께 저장한다.
  - 스토리지에 저장시 row 아 아니라 column 으로 나누어서 저장하여, 읽기시 지정한 컬럼의 데이터만 읽어가도록 하는 방식
  - 쿼리에 사용되는 열만 읽으면 되기에 메모리 사용 절감이 가능
- Column Compression
  - 디스크 처리량을 더 줄이기 위한 압축
  - Bitmap encoding 을 통해 압축이 가능해짐:
    - column의 고유 값 수(n)가 row의 수(m) 비해 적을 경우, n개의 고유 값 각각에 대한 비트맵을 만들어서 그 값이 1이면 맞고 아니면 0 
  - 단점: 쓰기 성능 저하
    - 압축된 컬럼 중간에 행 삽입 -> 모든 컬럼 파일 제작성 필요.
      - 극복: LSM-Tree 적용
        - 멤테이블에 쓰고 이후 컴팩션을 통해 컬럼파일로 대향의 병합 

### Aggregation: Data Cubes and Materialized Views (집계: 데이터 큐브와 구체화 뷰):

#### Materialized View:
- **Virtual View**: SQL 엔진이 쿼리 실행 시 기존 쿼리로 확장해서 처리하는 "바로가기"
- **Materialized View**: 쿼리 결과의 **실제 사본**을 디스크에 저장
  - 원본 데이터 변경 시 업데이트 필요 (비정규화된 복사본이므로) → 쓰기 비용 증가
  - OLTP에서는 잘 안 쓰지만, **읽기 중심의 데이터 웨어하우스에서는 유용**

#### Data Cube (OLAP Cube):
- 구체화 뷰의 특수 형태
- 여러 차원(날짜, 상품, 매장, 프로모션, 고객 등)으로 그룹화한 집계 값의 격자
  - 예: 날짜 × 상품 → 각 셀에 매출 합계
  - 행/열 방향으로 집계하면 차원을 하나 줄인 요약
- **장점**: 특정 쿼리(예: 어제 매장별 총매출)가 **사전 계산**되어 매우 빠름
- **단점**: 원시 데이터 쿼리만큼의 유연성이 없음 (큐브에 없는 차원으로 분석 불가)
- 따라서 대부분의 데이터 웨어하우스는 **원시 데이터를 최대한 유지**하면서, 데이터 큐브는 특정 쿼리의 성능 부스터로만 사용

### Questions Resolved After Reading the book & Further questions
- 디스크 해시 맵은 성능이 좋지 않음
  - 왜? -> random access I/O 가 너무 많아지고, 디스크가 찼을 때 확장 비용이 비싸며 해시 충돌 해소를 위한 로직이 필요해짐
- Log Structured Storage — Transaction Processing — B-Trees 의 엮임:
  - **트랜잭션 처리(OLTP)** 라는 영역 안에 두 가지 스토리지 엔진 학파가 존재:
    - **Log-Structured 학파**: append-only → Bitcask, SSTable, LSM-Tree, LevelDB, Cassandra, HBase, Lucene
    - **Update-in-place 학파**: 고정 크기 페이지 덮어쓰기 → **B-Tree** (모든 주요 관계형 DB + 많은 비관계형 DB)
  - 둘 다 OLTP용 인덱싱 구조이며, 서로 다른 이점을 가짐 (쓰기 vs 읽기)
- 분산화된 데이터 베이스에서의 데이터 웨어하우스는 어떻게 구현을 할까?
  - 이 읽기 작업 자체가 엄청난 부하 일 것 같은데 이를 어떻게 해결하는지?
    - (가용할 수 있는 방법중 하나 정도로 생각하자) CDC를 이용해 전달한 bin 로그를 이용해 발생한 내역을 데이터 웨어하우스에서 역재생
  - 웨어하우스는 당연히 분산을 기초로 두기에 분산 저장하는게 기초이고, 이를 위한 별도의 시스템이 필요할 것 같다.
