B-tree 에서는 트리에 잠금을 포함한다던데 이게 무슨 차이가 있을까?----각박한 현실속에 데이터 시스템에는 여러 문제가 생길 수 있음:1. 데이터베이스 소프트웨어/하드웨어는 연산 중에든 아니든 언제든 실패할 수 있다2. 연산 도중인 어플리케이션도 맛이 갈 수 있다3. 네트워크 단절(어플리케이션-디비 혹은 디비 노드간)도 언제든 갈 수 있다4. 여러 쓰기로 인해 쓰기가 덮어써질 수도 있다5. 부분적으로만 갱신된 데이터를 클라에 넘길 수도 있다6. 경쟁조건으로 인한 버그가 있을 수 있다=> 이러한 문제들을 단순화하는 메커니즘이 **트랜잭션**- 트랜잭션이란?:  - 어플리케이션에서 여러 읽기/쓰기를 하나의 논리적 유닛으로 묶는 방법  - 전체가 성공(commit)하거나 전체가 실패(rollback)## The Slippery Concept of a Transaction- 관계형 비관계형 가리지 않고 트랜잭션을 지원함  - 대부분은 40년전의 원형인 IBM의 System R의 트랜잭션의 기본 아이디어를 활용한다고 함- 2000년대 중반, 비관계형 디비(NoSQL)이 득세하며 [복제(replication)](chapter05.md)와 [파티셔닝](chapter06.md)이 기본이 되면서, **_트랜잭션을 아예 버리거나, 원래의 것보다 더 약한 보장을 하는 신세대의 데이터베이스들이 나왔다._**  - 그러나 이는 진실이라 볼 수 없고, 당연하게 트랜잭션도 이점과 한계가 있고 트레이드 오프를  이해해 어떤 상황에 활용하고 어떤 상황에 활용하지 않을지를 고려해야한다### The Meaning of ACID- A(Atomicity) C(Consistency) I(Isolation) D(Durability)- 데이터베이스에서 내결함성 메커니즘을 나타내는 용어- 그러나 디비마다 ACID의 구현은 다 제각각 -> 특히 **Isolation의 의미가 모호**  - 디비가 ACID를 준수(compliant)한다 하더라도 얼마나 기대할 수 있는지는 모호함#### Atomicity- 원자성은 문맥에 따라 다른 의미를 지님:  - 멀티 스레스 프로그래밍에서의 원자성: 연산의 동시성 보장    - ACID에서는 격리성(Isolation) 에서 이 내용을 다룸  - ACID 에서의 원자성: 트랜잭션의 원자성    - 트랜잭션으로 묶인 쓰기가, 어떠한 이유의 결함/장애등이 발생했다면 중단처리(abort)하여 없던 일로 만드는 것    - 다같이 성공 하거나 다같이 실패- 사실상 Abortability 가 더 적절한 단어이나 Atomicity 로 굳어짐#### Consistency- 문맥에 따라 다른 의미가 됨:  - ACID에서의 일관성: invariants(불변식)가 항상 지켜짐에 대한 보장- 그러나 이 invariants의 정의는 어플리케이션의 책임  - 디비가 invariant를 위반하는 잘못된 데이터의 쓰기를 막지는 못함   - 외래 키 조약, 유니크 키 등 디비에서의 invariants를 강제할 수도 있기는 하다#### Isolation- ACID에서의 격리성: 트랜잭션들이 서로 격리됨  - 트랜잭션은 다른 트랜잭션에 개입할 수 없다  - 여러 요청을 통해 하나의 row를 수정한다 하더라도 마치 순차적으로 실행됐을 때의 결과와 동일하도록 보장  - 트랜잭션A에서 여러 쓰기(write => INSERT, UPDATE, DELETE) 요청이 일어나면, 다른 트랜잭션B 에서는 모든 쓰기가 성공했거나 모든 쓰기가 실패해야함 #### Durability- ACID에서의 지속성: 성공적으로 커밋된 데이터는 하드웨어 결함이나 디비가 크래시 되더라도 없어지지 않는다는 보장- 싱글 노드라면: HDD나 SDD에 저장되었음을 의미- 복제된 디비라면: 최소로 정한 수의 노드들에 저장되었음을 보장- 지속성을 보장하기위해 IO나 복제가 완료될때 까지 트랜잭션을 열어두고 기다려야함### Single-Object and Multi-Object Operations- multi-object transactions -> 여러 데이터를 동기화시켜야 할 때 사용됨  - ex: (pg.229) 이메일을 수신하면 수신한 유저의 메일박스의 unread 카운터를 1 올려야 함.     - email에 대한 INSERT와 mailbox에 대한 UPDATE가 동시에 일어나야함  - TCP커넥션 안에서 BEGIN TRANSACTION 문 부터 ... COMMIT 문 까지를 하나의 트랜잭션의 범위로 취급    - 그러나 이는 이상적이지 않다고 한다.       - 클라이언트가 커밋을 보냄 -> 서버가 커밋 여부를 알려주기 전에 연결이 끊기면 클라는 커밋됐는지 아닌지 알 수가 없음.- Nonrelational Databases (NoSQL) 은 연산들을 그룹화할 수 없음#### Single-object writes- 단일 객체 쓰기에 대한 원자성(Atomicity) 보장:  - 크래시 리커버리를 위한 로그로 구현됨- 단일 객체 쓰기에 대한 격리성 (Isolation) 보장:  - 락을 통해 구현됨- 단일 객체에 대해 좀더 복잡한 atomic (isolated/serializable) 연산을 지원하는 디비들이 있다  - ex: increment 연산, compare-and-set (CAS) 연산 등  - 멀티 스레드 환경에서 동시다발적으로 쓰기 요청이 올떄 처리에 필요함  - 그러나 이 연산들은 트랜잭션으로 묶이는 연산이 아님#### The need for multi-object transactions- 분산 데이터베이스에서는 다중 객체 트랜잭션 지원을 포기함   - 여러 파티션에 걸쳐서 구현이 어려움 + 매우 높은 가용성과 퍼포먼스가 필요한 곳에 방해가 될 수도 있기 때문에 그렇다 함- 그러나 근본적으로 불가능한 것은 아니고, 이에 대한 구현은 [9장](chapter09.md) 에서 다룰 예정- 다중 객체 트랜잭션이 필요한 케이스들:  - 관계형/그래프 디비 경우의 외래키 참조가 유효한 상태이도록 보장할 수 있다    - ex: 참조를 가지는 여러 객체를 한번에 커밋해 참조를 적용  - 문서형 디비에선 역정규화(denormalization)이 흔하게 발생함 -> 역정규화된 여러 문서를 한번에 업데이트해야 싱크가 안맞는 문제를 예방할 수 있음  - 보조 인덱스 (secondary index)가 적용되는 경우라면 인덱스들에 대한 변경도 트랜잭션 안에서 일괄적으로 일어나야 특정 색인에서는 검색이 되고 다른 색인에서는 수정된 내용으로 검색이 안되는 상황을 막을 수 있다    - ex: email 과 name으로 색인이 걸려있는데 만약 둘 다 수정되는 경우에 한쪽의 인덱스가 수정이 되지 않는 경우#### Handling errors and aborts- 트랜잭션의 키 피쳐: abort 되고, 안전하게 제시도 될 수 있다는 점  - 만약 디비가 원자성, 격리성, 지속성을 지키지 못하는 상황이 발생할 것 같으면 트랜잭션을 전체 폐기하여 어중간한 상태를 방지- 그러나 모든 시스템이 이 키 피쳐를 적용하진 않음  - 리더리스 복제를 지원하는 디비들은 best effort 기반으로 동작하고, 에러로부터의 복구를 어플리케이션의 책임으로 취급함- 그러나 abort된 트랜잭션의 재시도는 문제가 되는 경우가 있다  1. 트랜잭션은 성공했으나 네트워크 이슈로 ack를 못받은 경우, 재시도를 하게 되면 불필요한 데이터가 생길 수 있다  2. 데이터베이스가 너무 과부하로 너무 힘들어서 에러가 발생한 경우라면 재시도는 상황을 악화시킬 것  3. 일시적인 오류 (데드락, 격리성 위반, 일시적 네트워크 단절, 페일오버) 의 **경우에만** 재시도가 의미가 있고, 영구적인 (constraint violation, 디비 서버 완전 죽음 등)의 경우에는 재시도는 의미가 없다  4. 트랜잭션의 실행으로 인한 부수 효과로 외부에서 처리되는 로직이 있다면 부수 효과가 불필요하게 실행되는 경우가 존재함.     - ex: 이메일을 보내고 데이터를 저장하는 시스템에서 트랜잭션을 재시도 할 때 이메일이 새로 보내지면 안됨       - 이러한 문제는 2페이즈 커밋을 활용해 해결 가능함 ([9장 354쪽에서 다룰 예정](chapter09.md))## Weak Isolation Levels- **동시성 이슈 (레이스 컨디션)은 하나의 트랜잭션이 다른 트랜잭션에서 수정되고 있는 데이터를 읽으려고 하거나, 두개의 트랜잭션이 동시에 하나의 데이터를 수정하려 할 때만 발생한다.**- 디비는 어플리케이션 개발자들로부터 동시성 이슈를 숨기기 위해 *트랜잭션 격리*를 제공함  - **_Serializable 레벨에선 마치 동시성이 없는 것 (serially 적용되어 순차적으로 적용) 처럼 프로그래밍할 수 있게 해줌_**    - 그러나 serializable은 퍼포먼스 이슈가 있음### Read Commited- 가장 기본적인 격리 수준- 두가지를 보장함:  - No dirty read:    - 읽을 때 커밋된 내용만 읽음  - No dirty write:    - write할 때 커밋된 내용만 커밋- dirty read 란?  - 다른 트랜잭션이, 아직 커밋되지 않은 데이터를 볼 수 있는 경우- dirty read를 방지하는게 유용한 이유:  - 부분적으로 수정된 내용을 볼 수 있게되면 혼란이 있을 수 있음    - ex: 7-2의 이메일과 안읽은 이메일 수의 경우처럼 이메일은 받았는데 안읽은 이메일 수는 증가하지 않을 수 있음  - 커밋된 적 없고 곧 롤백될 데이터가 유저에게 전송될 수 있음- Read Committed는 [Lost Updates문제는 해결하지 못함](#preventing-lost-updates)- read committed 의 구현:  - dirty write 방지:    - row-level lock을 활용해 하나의 트랜잭션에서 write lock을 가지고 있으면 그 트랜잭션이 종료될 때까지 다른 트랜잭션들은 락을 점유 할 수 없고 대기해야함  - dirty read 방지:    - read lock을 가져가는 방법은 read-only 트랜잭션에 대해 응답 시간 지연의 원인이 되고, 락을 기다리다 어플리케이션에 연쇄적인 악영향을 끼칠 수 있기 때문    - 그래서 대부분의 데이터베이스에선, write lock을 지닌 트랜잭션의 이전값과 현재값을 다 가지고 있고, 아직 쓰기 락을 가진 트랜잭션이 진행중이면 다른 트랜잭션에서는 이전 값을주고, 쓰기 락을 가진 트랜잭션이 커밋되면 새값을 주는 방식으로 구현      - 단순히 커밋된 값 + 아직 커밋안된 수정된 값만 들고 있는게 아니라 MVCC를 통해 스냅샷을 관리한다 ### Snapshot Isolation and Repeatable Read- Read Committed 에서 발생하는 문제:  - nonrepeatable read / read skew:    - 하나의 트랜잭션에서 같은 row의 값을 2번 읽을 때 (SELECT) 값이 달라지는 현상       - ex: 두개의 트랜잭션이 있고, 같은 데이터는 읽는 트랜잭션과 쓰는 트랜잭션이 있을 때, 쓰는 트랜잭션 쪽이 먼저 끝났고, 읽는 트랜잭션에서 같은 값을 두번 불러오게 되는 경우 Read Committed 레벨에서는 값이 달라진다    - 이런 일시적인 비일관성이 용인되지 않는 경우가 존재한다    - backup & analytic query and integrity checks- Snapshot Isolation이란?  - 트랜잭션에서 데이터베이스의 _consistent snapshot_ 을 읽고, 트랜잭션이 종료될 때까지 그 값만 읽어 온다    - consistent snapshot이란?      - 트랜잭션 시작시의 커밋되어 있던 데이터- 스냅샷 격리의 구현  - dirty write 방지는 Read Committed와 같이 row level write 락을 사용함  - read는 아무런 락이 존재하지 않음.    - _readers never block writers, and vice versa_    - **_MVCC (Multi-Version Concurrency Control)_**      - 하나의 row 데이터 (객체)를 위해 여러가지 커밋된 버전을 기록하고, 각각의 트랜잭션이 해당 트랜잭션의 시점에 맞는 버전을 사용하는 방식      - MVCC 기반 스냅샷 격리의 구현 (PostgreSQL에서):        - 트랜잭션의 고유 아이디 (txid) 를 기준으로 생성시와 삭제시에 해당 txid를 기록, 수정이 발생할시 생성과 삭제로 치환해 해당 버전을 기록하여 스냅샷으로 활용         - 각각의 객체는 created_by와 deleted_by 를 가지며, 이 값들은 트랜잭션 iD로 채워짐        - 삭제 처리는 deleted_by 에 트랜잭션 id가 생길때 발생하며 소프트 딜리트로 구현됨        - 이후 어떠한 트랜잭션에서도 삭제 처리된 데이터를 액세스하지 않으면 garbage collecting process에서 삭제처리된 객체들을 데이터베이스에서 삭제하고 공간을 확보함<a id="visibility-rules"></a>- 일관된 스냅샷을 보기위한 visibility rules  - txid를 이용해 어떤 버전의 객체를 볼지, 어떤 객체가 보이지 않는지를 결정함  - 두가지 컨디션이 통과하면 트랜잭션에서 해당 버전의 객체를 볼 수 있음    1. 읽는 쪽의 트랜잭션의 시작시, 해당 버전의 객체를 생성한 트랜잭션이 커밋됨    2. 읽는 쪽의 트랜잭션 시작시에 해당 버전의 객체가 삭제되지 않았거나, 삭제하는 트랜잭션이 커밋되지 않았음  - 객체의 값을 절대 수정하지 않고 새 버전을 계속 생성하는 것으로 consistent snapshot을 만들 수 있다- indexes and snapshot isolation  - 버전들을 한 인덱스에 한번에 두고, 읽을 때 현재 트랜잭션에 보이는 버전만 골라내는 방식    - 인덱스는 많이 보여주고, MVCC 의 visibility rule로 거르는 방식  - B-Tree를 자체 버전 별로 복사해가며 각 시점마다 별도의 루트를 가지게 하는 방식- Naming confusion  - Oracle은 Serializable이 부름  - MySQL & PostgreSQL은 Repeatable Read 라고 부름  - 왜 다 부르는 이름이 다르냐? -> SQL 표준에 스냅샷 격리의 개념이 없음.    - 대신 repeatable read는 존재했고, 이 개념이 스냅샷 격리와 비슷해 MySQL & PostgreSQL이 이를 사용한 듯 하다.  - SQL 표준의 격리 수준 정의는 모호하여, 데이터베이스들마다 repeatable read 로 제공하는 격리 수준이 다 다른 문제가 있다.### Preventing Lost Updates- Read Committed & Snapshot Isolation -> 동시 쓰기가 있을 때 읽기 전용 트랜잭션이 어떤 것을 볼 수 있냐에 대한 보장 (dirty write는 쓰기에 관한 내용)- lost update (갱신 손실) 이란?  - 같은 데이터에 대해 read-modify-write 사이클을 가진 두개의 트랜잭션이 돌 때, 하나의 트랜잭션에서의 수정이 손실(lost) 되는 현상    - ex: 위키 페이지 수정을 동시에 진행시 마지막 수정만 적용됨  - 해결 방안:    - Atomic Write Operations      - read-modify-write 사이클 없이, 하나의 쿼리에서 업데이트를 할때 read시점의 조건문을 이용해 표현하는 방법.      - 대부분의 관계형 디비에서 concurrency-safe 함      - exclusive lock 을 획득해서 구현됨      - ORM을 쓰면 원자적 연산을 쓰기보다 불안전한 read-modify-write 사이클을 실행하는 코드를 짜기 쉽다    - Explicit Locking (명시적 락킹)      - SELECT .. FOR UPDATE 를 이용해 명시적인 락을 걸어 다른 트랜잭션들을 기다리게 만드는 방법      - read-modify-write 사이클을 사용가능      - 주의해서 사용하지 않으면 race condition을 유발한다    - Automatically detecting lost updates      - 데이터베이스의 트랜잭션 매니저를 통해 lost update을 감지시키고 트랜잭션을 abort 하고 read-modify-write 사이클을 재시도       - 스냅샷 격리와 매우 효율적으로 수행이 가능함        - MySQL의 repeatable read는 감지를 못함. 어떤 이들은 이를 근거로 MySQL이 스냅샷 격리를 지원하지 않는다고 주장      - 어플리케이션 코드에서 디비의 특수 기능을 쓸 필요가 없고 자동으로 lost update가 감지되어 오류가 덜 발생하게 도와줌      - 그러나 락이나 원자적 연산을 쓰는 것을 잊게 해 버그를 유발할 수도 있다    - Compare-and-set      - 마지막에 읽은 시점의 값이 변하지 않았을때만 수정을 하도록 하기위한 원자적 연산      - 이전 스냅샷을 WHERE 절을 통해 읽어 올 수 있다면 방지 할 수 없게된다- Conflict Resolution and Replication  - 다시 읽고 돌아와 보자### Write Skew and Phantoms- Write Skew  - 동시의 (concurrent) 여러 트랜잭션이 각각 다른 row를 최신화 되지 않은 read 를 기반으로 invariant (비지니스 룰 등)을 만족함을 확인 후 수정하였는데 invariant가 깨진 (비지니스 룰이 깨진) 상황    - Serializable isolation level 을 적용하여 해결이 가능함    - 명시적 락킹을 활용하는 방법이 차선- Phantom  - 어떤 트랜잭션에서 실행한 쓰기가 다른 트랜잭션의 검색 질의 결과를 바꾸는 효과  - predicate 이 어떻게 되냐에 따라 잠글 대상이 없는 경우가 존재해 락을 통한 해결이 불가능함  - 이를 해결하는 방법중 하나로, materializing conflict (충돌 구체화) 를 통해 락의 획들을 위한 테이블을 설계해 락을 획득하여 팬텀을 방지 가능    - 그러나 이는 어렵고 에러에 취약해지는 문제가 있음      - 정말 최후의 보루로 사용하는 방법      - 보통은 serializable isolation level을 도입해 해결## Serializability (직렬성)- 가장 강력한 격리 수준- 여러 트랜잭션이 병렬로 실행되더라도, 하나씩 직렬로 (serially) 실행될 때와 같도록 보장함 -> **_데이터베이스가 모든 race condition을 막아줌_**### Actual Serial Execution- 한번의 하나의 트랜잭션만 하나의 쓰레드에서 실행시키는 방법- Redis, VoltDB, Datomic 에서 사용- 이 방법은 그나마 최신 (약 2007년 경) 부터 가능하다고 받아들여짐  1. 랩이 싸짐 -> 트랜잭션에 필요한 모든 데이터를 인메모리에 담아 트랜잭션의 실행을 (디스크에서 가져오는 것 보단) 훨씬 빠르게 만들 수 있음  2. OLTP에선 짧고 간결한 read & write가 주를 이룸- 락킹에 의한 오버헤드가 없어 동시성 문제에 대해 더 빠름- 그러나 처리량이 하나의 CPU 코어에 제한됨  - 최대한의 활용을 위해선 전통적인 트랜잭션의 형태와는 다른 구조를 필요로함- **Encapsulating transactions in stored procedures**   - 프로시져를 이용해 원자적 쿼리를 만드는 방법  - 여러 인터랙티브한 트랜잭션을 허용하지는 않으나, 그를 대체하는 기능을  - read-modify-write 사이클을 통해 어플리케이션에서 검증 후 수정 요청을 보내는게 아닌, read 할 컨디션을 미리 정해둔 프로시져를 활용해 수정까지 하나의 트랜잭션에서 해버리는 방식  - 단점:    - 디비 벤더마다 각각의 프로시져 언어를 사용함 -> 현재의 시점에선 어렵고 읽기 쉽지 않음.      - 그러나 VoltDB는 자바/그루비를 Datomic은 자바/클로저를 그리고 Redis는 Lua를 사용하는 등 현대화되고 있다함    - 데이터베이스에서 돌아가는 코드는 관리가 어려움    - 디비는 어플리케이션 코드보다 성능이 중요함 (여러 어플리케이션 서버가 붙을 수 있기에) -> 리소스를 많이 잡아먹는 프로시져로 인해 트래블이 생기기 쉬움 (어플리케이션 코드보다 훨씬 더!)  - 그러나 프로시져와 인메모리 데이터가 있기에 싱글 스레드에서 모든 트랜잭션을 실행하는게 가능해짐    - I/O 대기가 없음    - 동시성 제어 메커니즘의 오버헤드도 회피됨    - 상당히 좋은 처리량 겟!  - 파티셔닝과 단일 쓰레드 실행:    - 단일 쓰레드의 한계를 극복하기 위한 방안 중 하나    - 파티셔닝으로 여러 쓰레드에서 각 파티션에 프로시져를 각각의 쓰레드에서 실행시키는게 가능해짐    - 그러나 만일 여러 파티션을 전부 포괄해야하는 프로시져의 경우 모든 파티션에 대해 락을 획득해야한다. -> 추가적인 coordination 오버헤드### Two-Phase Locking (2PL)- Two-Phase Commit(2PC) 와 굉장히 유사한 이름이나 다른 개념 (2PC는 [9장](chapter09.md)에서 다룰 예정)- 트랜잭션의 시작과 함께 락을 획득하고 (페이즈 1) 트랜잭션의 종료에 락을 릴리즈한다 (페이즈 2) -> 이래서 2PL이라 불림- MySQL & SQL Sever 의 serializable isolation 이 2PL을 사용함- 읽기는 동시 접근이 가능하지만, 수정에 대해서는 읽기를 진행중인 다른 트랜잭션이 모두 종료되었을 때에만 락을 획득할 수 있는 락  - 2PL에서는 이전 버전의 객체를 읽는 것이 허용되지 않음- 책에서 이 시점까지 다룬 lost update 나 write skew 등 모든 race condition으로 부터 보호해줌- **Implementation of 2PL**  - 읽어오는 시점에 exclusive lock 을 가진 다른 트랜잭션이 없다면 shared mode lock 을 획득해 읽어 올 수 있음  - 읽어오려는 시점에 다른 트랜잭션이 exclusive 락을 가졌다면 그 트랜잭션이 종료될때까지 대기한 후 종료되었을 때 shared mode lock 을 획득해 데이터를 가져옴  - 트랜잭션 내부에서 읽어온 후 수정/삭제/생성을 행하려는 경우, shared mode lock을 exclusive lock으로 업그레이드해야함     - -> 업그레이드되는 도중 만약 다른 트랜잭션에서 쓰려는 대상에 대한 exclusive lock 을 가진 경우는 대기- **_락이 많이 발생하고, 다른 트랜잭션의 락을 기다리다 죽는 경우도 있다 -> 이게 deadlock_**  - DB는 자동으로 데드락을 감지하고, 트랜잭션 하나를 abort해 데드락을 해제시킴    - abort된 트랜잭션은 어플리케이션에서 재시도를 책임져야함- **2PL Performance**  - 다른 격리 레벨에 비해 성능(트랜잭션 처리량과 쿼리의 응답 시간)이 매우 좋지 않다.    - read committed도 데드락이 발생할 수 있으나, 2PL serializable isolation에선 훨씬 더 자주 발생한다.#### Predicate Lock  - shared/exclusive lock 과 비슷하게 특적 오브젝트에 락을 거나, 하나가 아닌 검색조건 (where절)의 포함되는 모든 데이터를 대상으로 락을 건다    - TX1 에서 시간 범위로 SELECT 를 하면 shared-mode predicate 락을 가져옴. 그러나 만약 그 범위의 데이터들에 대해exclusive 락을 가진 TX2가 존재한다며 TX2의 락이 풀리기를 기다려야한다    - TX1 에서 값을 생성하거나 수정 삭제할 경우, 그 범위의 대한 predicate 락이 존재하는지 확인 -> 존재할 경우 풀릴때까지 대기해야함  - 2PL이 predicate lock을 포함하면 모든 종류의 write skew와 레이스 컨디션을 방지하여 진짜 serializable 해진다   - 동시 요청, 범위 질의에 대해 락을 계속 기다려야해 성능이 안좋다. (기다리는 케이스가 너무 많기에)#### Index-Ranged (next-key) locks- Predicate lock 의 성능이 안좋기에, 이를 대체하는 락  - 대부분의 데이터베이스들은 predicate 락 대신 index-range 락을 사용한다고 함- 질의 범위로 락을 거는게 아니라, 인덱스에 락을 걸어, 해당 인덱스 범위에 있는 데이터들에 대한 접근을 막는 방식- 만약 인덱스로 지정할만한 값이 없다면 **테이블에 락을 건다.** 이는 인덱스 범위 락의 이점을 없애고 성능 저하의 원인이 될 수 있다. - 근데 이게 왜 overheads더 낮을 수 있는걸까?  - 두 락 모두 락을 적용해 대기시키위해서는 현재 트랜잭션의 데이터가 영향받는지 매칭을 계속해서 확인을 해야한다.  - 이 질의 자체가 인덱스가 되면 값이 정렬되어 있고, 특정 범위에만 들어가 있는지를 확인하는 것이 빨라진다.  - predicate 락의 경우, 트랜잭션 마다 범위가 제각각일 수 있기에 매번 검사를 위해 아래와 같은 질문들을 계속해야한다    - “이 조건과 저 조건이 겹치나?”     - “이 insert가 저 predicate를 만족하나?”     - “아직 없는 row까지 포함하면 어떤 공간을 막아야 하나?”  - index-range 락의 경우 인덱스의 특정 키, 범위를 찾는게 빠르고, 그 구간에 락이 있는지를 확인하면 되서 빠르다.    - 대신 범위가 더 넓어지는 문제가 있고, 이 때문에 불필요하게 락을 점유하는 경우가 있을 수 있을 것 같다.### Serializable Snapshot Isolation (SSI)- Serializable 한 격리도 필요하지만 2PL은 너무 느리다.. 이를 극복할 방법으로 가장 최신의 방식이다!  - snapshot 격리와 비교해 조금의 성능 손해만 있다고 한다  - 작가는 미래에는 새로운 default 가 될 것 같다고 예측- 2PL은 비관적 **_(pessimistic) 한 동시성 컨트롤 메카니즘_**  - 조금이라도 뭔가 잘 못 될 것 같음 -> 잘못될 일이 없어질 때까지 기다리자!- SSI는 낙관적 **_(optimistic) 한 동시성 칸트롤 메카니즘_**  - 커밋 시점에 뭔가 잘 못 되었다 -> abort and retry!  - 하나의 row를 여러 트랜잭션에서 접근하면 (hot key) 너무 많은 트랜잭션이 abort 되어야하는 문제가 있음- SSI는 snapshot isolation + 직렬화 충돌을 감지해 어떤 트랜잭션을 abort 할지를 정하는 알고리즘  - 어떻게 변화를 감지하는가? -> MVCC!!    - MVCC를 이용해 오래된 읽기 감지      - ex: TX1,2에서 각각 같은 데이터를 SELECT 하고 TX1가 먼저 커밋에 성공함 -> TX2는 실패해야한다.        - TX2가 커밋하려는 시점에 [MVCC visibility rule](#visibility-rules) 에 따라무시하고 있던 변경 사항이 적용되었는지 확인 -> 적용되었으면 abort        - 만일 TX1이 read-only 트랜잭션이었다면 write skew 걱정이 없음!    - MVCC를 이용해 이전에 읽어온 값에 대한 쓰기 감지      - 인덱스 범위 락과 비슷하게 검색 질의에 사용된 인덱스에 대해 일종의 트래킹 인포를 저장하고, 각 트랜잭션에 해당 트래킹 인포에 감지되는 데이터가 수정되는 경우에 그 트랜잭션으로 알림을 보낸다. 이후 실제 커밋 시점에 확정되었는지 검사한 후 노티받은 데이터에 대한 커밋이 이뤄져 있으면 그 트랜잭션을 abort        - “어떤 트랜잭션이 어떤 범위/조건을 읽었는지”를 인덱스 엔트리나 테이블 수준에 잠시 기록해 두고, 나중에 어떤 트랜잭션이 write 할 때** 그 기록을 조회해서 “아, 이 write가 누군가의 prior read를 깨뜨릴 수 있겠구나”라고 표시하여 다음 write들을 abort 시키는 구조- SSI의 성능적 트레이드 오프:  - 각 트랜잭션에서 발생하는 읽기 & 쓰기를 트래킹하기 때문에 만약 디테일한 정보를 트래킹한다면 이에 대한 오버헤드가 엄청 커질 수도 있다.    - 트래킹하는 정보를 덜 디테일하게 만들면 빠를 수는 있으나 반드시 필요한 것 이상의 불필요한 abort 들이 발생할 수 있다.  - serial execution과 비교하면 하나의 머신의 CPU 를 전부 사용할 수 있는 장점이 있다.  - 분산 파티션된 환경에서도 여러 파티션들에 대해 읽기 쓰기 작업에서 직렬 격리 (serializable isolation)을 보장할 수 있다.    - FoundationDB의 detection은 여러 머신에서 굉장히 높은 처리량을 보여준다 함  - 고려해야할 사항:    - abort되는 비율이 SSI의 성능을 좌우하는 키 팩터      - 그렇다하더라도 2PL이나 단일 쓰레드 serial execution 보다는 덜 영향 받음    - 트랜잭션이 오래 열리지 않는 환경에서 사용하는 것이 베스트      - 트래킹 정보가 불필요하게 오래 남게되며 abort되는 트랜잭션이 늘어날 확률이 높아지기 때문- 실제 SSI를 적용가능한 데이터베이스: PostgreSQL, FoundationDB
