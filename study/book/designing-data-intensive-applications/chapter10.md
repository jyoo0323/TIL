# **Ch.10 Batch Processing**

---

- 책의 Part I, II 에서는 **요청(request)과 질의(query)** 그리고 **응답(response)과 결과(result)** 에 대해 다루었다
  - 데이터베이스, 캐시, 검색 인덱스, 웹 서버 등이 모두 이 방식으로 동작함
  - 이런 **온라인 시스템**은 사용자가 응답을 기다리고 있으므로 **응답 시간(response time)** 이 핵심 관심사
- 그러나 시스템을 구축하는 유일한 방법은 아니다. 저자는 **세 가지 유형의 시스템**을 구분한다:

### **1. Services (온라인 시스템)**

- 클라이언트의 요청이나 명령을 기다리고, 최대한 빨리 처리하여 응답을 보냄
- 주요 성능 지표: **응답 시간(response time)**, **가용성(availability)**

### **2. Batch processing systems (오프라인 시스템)**

- 대량의 입력 데이터를 받아 **작업(job)** 을 수행하고 출력 데이터를 생산
- 작업은 보통 수 분에서 수 일이 걸리며, 사용자가 완료를 기다리지 않음
- 주기적으로 실행되도록 스케줄링 (예: 하루에 한 번)
- 주요 성능 지표: **처리량(throughput)** -- 특정 크기의 입력 데이터셋을 처리하는 데 걸리는 시간

### **3. Stream processing systems (준실시간 시스템)**

- 온라인과 오프라인/배치 처리의 중간
- **근실시간(near-real-time)** 또는 **거의 실시간(nearline)** 처리라고도 함
- 배치 처리와 마찬가지로 입력을 소비(consume)하고 출력을 생산하지만, 이벤트가 발생한 직후에 처리
- 배치 시스템보다 **지연 시간이 낮음**
- 스트림 처리는 배치 처리 위에 구축됨 (Chapter 11에서 다룸)

---

- **MapReduce**는 2004년에 발표된 배치 처리 알고리즘으로, "Google을 엄청나게 확장 가능하게 만든 알고리즘"이라 불림
  - 이후 Hadoop, CouchDB, MongoDB 등 다양한 오픈소스 데이터 시스템에 구현됨
  - 이전 데이터 웨어하우스용 병렬 처리 시스템에 비해 꽤 저수준 프로그래밍 모델이지만, **범용 하드웨어(commodity hardware)** 에서 대규모 처리가 가능하다는 점에서 큰 진보
- 배치 처리의 역사:
  - 1890년 미국 인구조사에 사용된 Hollerith 기계(펀치카드 집계기)가 반자동화된 배치 처리를 구현
  - MapReduce는 1940~1950년대 IBM 카드 정렬 기계와 유사

## **Batch Processing with Unix Tools**

### **Simple Log Analysis**

- 웹사이트에서 가장 인기 있는 5개 페이지를 찾는 Unix 명령어 체인:

    ```bash
    cat /var/log/nginx/access.log |
      awk '{print $7}' |     # 7번째 필드(요청 URL) 추출
      sort |                 # URL 알파벳순 정렬
      uniq -c |              # 중복 제거 및 카운트
      sort -r -n |           # 숫자 기준 내림차순 정렬
      head -n 5              # 상위 5개만 출력
    ```

- 이 명령어 체인은 **수 기가바이트의 로그 파일도 수 초 만에 처리** 가능

### **Sorting versus in-memory aggregation**

- **Ruby 스크립트** (인메모리 해시 테이블): working set(랜덤 접근이 필요한 메모리 양)은 고유 URL 수에만 의존
- **Unix 파이프라인**: 해시 테이블 없이 정렬에 의존
- 어떤 접근이 더 나은가?
  - 고유 URL 수가 적은 소~중규모 웹사이트: 모든 URL과 카운터가 메모리에 들어가므로 인메모리 해시 테이블이 잘 동작
  - **working set이 가용 메모리보다 큰 경우**: 정렬 방식이 디스크를 효율적으로 사용할 수 있어 유리
    - SSTables와 LSM-Trees에서 논의한 것과 같은 원리: 데이터 청크를 메모리에서 정렬하고 디스크에 세그먼트 파일로 쓴 뒤, 여러 정렬된 세그먼트를 하나의 더 큰 정렬 파일로 병합
    - GNU Coreutils의 **sort** 유틸리티는 메모리보다 큰 데이터셋을 자동으로 디스크로 스필링하고, 여러 CPU 코어에 걸쳐 자동으로 정렬을 병렬화함

### **The Unix Philosophy**

- 로그 파일을 명령어 체인으로 쉽게 분석할 수 있었던 것은 우연이 아니다. Unix의 핵심 설계 사상이었다
- **Doug McIlroy** (Unix 파이프의 발명자, 1964년): "프로그램을 정원 호스처럼 연결하는 방법이 있어야 한다"
- 4가지의 원칙은 아래와 같다:
1. **각 프로그램이 한 가지 일을 잘 하도록 만들라.** 새로운 작업을 하려면 새로 만들고, 기존 프로그램에 "기능"을 추가하여 복잡하게 만들지 말라
2. **모든 프로그램의 출력이 아직 알려지지 않은 다른 프로그램의 입력이 될 것을 예상하라.** 불필요한 정보로 출력을 어지럽히지 말라
3. **소프트웨어를 일찍 시도해 볼 수 있도록 설계하고 구축하라** (이상적으로는 몇 주 이내). 서투른 부분은 과감히 버리고 다시 만들라
4. **프로그래밍 작업을 가볍게 하기 위해 미숙한 도움보다 도구를 사용하라**
- 이 접근법은 오늘날의 **Agile과 DevOps movements**와 유사하다

### **A uniform interface**

- 한 프로그램의 출력이 다른 프로그램의 입력이 되려면, 같은 데이터 형식을 사용해야 한다
- Unix에서 그 인터페이스는 **파일(file)** -- 더 정확히는 파일 디스크립터(file descriptor)
  - 파일은 단순한 **바이트의 순서 있는 시퀀스**
  - 이 단순한 인터페이스 덕분에 매우 다양한 것들이 같은 인터페이스로 표현 가능: 실제 파일, 프로세스 간 통신 채널 (Unix 소켓, stdin, stdout), 디바이스 드라이버, TCP 소켓 등
- 관례적으로 대부분의 Unix 프로그램은 바이트 시퀀스를 **ASCII 텍스트**로 취급
  - `\n`(newline)으로 구분된 레코드 목록으로 취급함
  - 모든 프로그램이 같은 레코드 구분자를 표준화했다는 사실이 상호 운용성을 가능하게 한다

### **Separation of logic and wiring**

- Unix 도구의 또 다른 특징: **표준 입력(stdin)** 과 **표준 출력(stdout)** 의 사용
- 프로그램이 stdin과 stdout만 사용하면, 쉘 사용자가 원하는 대로 입출력을 연결 가능
- 이것은 **느슨한 결합(loose coupling)**, **지연 바인딩(late binding)**, 또는 **제어의 역전(inversion of control)** 의 한 형태
- 입출력 배선을 프로그램 로직에서 분리하면 작은 도구들을 더 큰 시스템으로 합성하기 쉬워짐

### **Transparency and experimentation**

- Unix 도구가 성공적인 이유 중 하나: 무슨 일이 일어나는지 **쉽게 볼 수 있다**는 점
  - **입력 파일은 불변(immutable)으로 취급됨** -- 원본 파일을 손상시키지 않고 원하는 만큼 자주 실행 가능
  - **파이프라인을 어느 지점에서든 끝내고** `less`로 출력을 파이프하여 예상 형태인지 확인 가능
  - **파이프라인 한 단계의 출력을 파일에 쓰고** 그 파일을 다음 단계의 입력으로 사용 가능
- **Unix 도구의 가장 큰 한계**: 단일 머신에서만 실행된다는 것 -> 이를 극복하려고 Hadoop 같은 도구가 등장

## **MapReduce and Distributed Filesystems**

- MapReduce는 여러 분산된 파일 시스템 (ex HDFS) 에 나뉘어져 있는 큰 데이터 셋을 처리하기 위한 코드를 작성하기 위한 프로그래밍 프레임워크이다.
- Unix 도구와 비슷하지만, 잠재적으로 여러대의 **머신**에 분산되어 실행
- Unix 도구처럼 상당히 blunt & brute-force이지만 놀라울 정도로 효과적
- 단일 MapReduce 작업은 단일 Unix 프로세스에 비유: 하나 이상의 입력을 받아 하나 이상의 출력을 생산
- Unix 도구가 stdin과 stdout을 사용하는 반면, **MapReduce 작업은 분산 파일시스템에서 파일을 읽고 씀**

### **HDFS (Hadoop Distributed File System)**

- Google File System(GFS)의 오픈소스 재구현
- **Shared-nothing 원칙**기반
  - NAS/SAN의 shared-disk 접근 방식(중앙 집중형 스토리지, 특수 하드웨어)과 대조적
  - 특수 하드웨어 없이, 일반 데이터센터 네트워크로 연결된 일반 컴퓨터만 사용
- **HDFS 구성:**
  - 각 머신에서**데몬 프로세스**가 실행되어, 다른 노드가 해당 머신에 저장된 파일에 접근할 수 있는 네트워크 서비스를 제공
  - **NameNode**라는 중앙 서버가 어떤 파일 블록이 어느 머신에 저장되어 있는지 추적
  - 개념적으로 모든 머신의 디스크를 사용할 수 있는 **하나의 큰 파일시스템**을 생성
  - **내결함성**: 파일 블록이 여러 머신에 **복제(replicated)** 되거나 **삭제 코딩(erasure coding)** 사용
  - Erasure coding 이란?
    - 파일을 N개의 데이터 조각 + M개의 복구 조각으로 나눠 저장해서, 일부 조각이 사라져도 남은 조각들로 원본을 복구할 수 있게 하는 기술
    - 전체를 복제하는 건 비용이 너무 큼. 저장시 복구용과 원본을 좀 나누는 형식으로 저장해 일부에서 원본을 복구할 수 있도록 만드는 것.

### **MapReduce Job Execution**

- MapReduce의 데이터 처리 패턴은 4단계로 구성:
  1. **입력 파일 읽기 및 레코드 분리**
  2. **Mapper 함수 호출** -- 각 입력 레코드에서 키와 값을 추출
  3. **키-값 쌍을 키 기준으로 정렬** -- 이 단계가 **암묵적(implicit)** 이므로 직접 작성할 필요 없음
  4. **Reducer 함수 호출** -- 정렬된 키-값 쌍을 순회하며 같은 키의 값들을 병합
- **Mapper**: 모든 입력 레코드에 대해 한 번씩 호출되며, 키와 값을 추출. **상태를 유지하지 않으며** 각 레코드는 독립적으로 처리
- **Reducer**: 같은 키에 속하는 모든 값을 모아 iterator와 함께 호출. 출력 레코드를 생성

### Distributed execution of MapReduce

- **"데이터 가까이에서 연산 수행(putting the computation near the data)"** 원칙: MapReduce 스케줄러는 입력 파일의 복제본이 저장된 머신에서 mapper를 실행하려고 시도하여, 네트워크로 파일을 복사하는 것을 막아 부하를 절감 + locality 증가
- Mapper가 출력을 마치면 스케줄러가 reducer에 알려 mapper의 출력 파일을 가져오게 함
  - 파티셔닝, 정렬, 데이터 복사 과정을 "shuffle"이라 부름
  - Map 태스크 수는 입력 파일 블록 수로 결정, **reduce 태스크 수는 작업 작성자가 설정**
  - 같은 키의 모든 키-값 쌍이 같은 reducer에 도달하도록 **키의 해시값**을 사용하여 배정
- **정렬 방식**: 각 map 태스크가 키의 해시를 기반으로 출력을 reducer별로 파티셔닝하고, 각 파티션을 mapper의 로컬 디스크에 정렬된 파일로 기록 (SSTables/LSM-Trees 기법과 유사)
- Reduce 태스크는 mapper들의 파일을 가져와 **정렬 순서를 유지하며 병합(merge)**
  - 출력 레코드는 **분산 파일시스템(HDFS)** 의 파일에 기록

### **MapReduce Workflows**

- 단일 MapReduce 작업으로 해결할 수 있는 문제의 범위는 **제한적**
  - MapReduce 작업을 **워크플로우로 연결(chaining)** 하는 것이 매우 일반적
    - 한 작업의 출력이 다음 작업의 입력이 됨
- Hadoop MapReduce 프레임워크 자체에는 워크플로우 지원이 없음
  - **디렉토리 이름으로 암묵적으로 연결**: 첫 번째 작업이 HDFS의 지정 디렉토리에 출력, 두 번째 작업이 같은 디렉토리를 입력으로 읽음
- 작업 간 의존성을 관리하기 위해 워크플로우 스케줄러가 개발됨: **Oozie, Azkaban, Luigi, Airflow, Pinball**
  - 50~100개의 MapReduce 작업으로 구성된 워크플로우가 흔함
- 더 높은 수준의 도구: **Pig, Hive, Cascading, Crunch, FlumeJava** -- 여러 MapReduce 단계의 워크플로우를 자동으로 구성

### **Reduce-Side Joins and Grouping**

- **조인(Join)의 필요성**
  - 많은 데이터셋에서 하나의 레코드가 다른 레코드와 연관(association)을 가짐
    - 관계형 모델의 **외래 키(foreign key)**, 문서 모델의 **문서 참조**, 그래프 모델의 **엣지(edge)**
  - MapReduce에는 인덱스 개념이 없음 -> 입력 파일 전체를 읽음 (full table scan)
    - 분석 쿼리에서는 대규모 레코드에 대한 집계 계산이 일반적이므로 전체 스캔이 합리적

### **Example: analysis of user activity events**

- 사용자 활동 이벤트 로그(activity events)와 사용자 데이터베이스를 조인해야 하는 경우
  - 이 구조는 **스타 스키마(star schema)** 와 유사함
    - 이벤트 로그가 팩트 테이블, 사용자 DB가 차원(dimension)
- 단순 구현 (레코드마다 원격 DB 조회)의 문제:
  - **왕복 시간(round-trip time)** 에 의해 처리량이 제한 → 하나 레코드 마다 원격 요청이 됨
  - 대량의 병렬 쿼리가 데이터베이스를 **overwhelm** 할 수 있음
    - 처리량이 한계를 넘어 지연과 실패가 발생

### **Sort-merge joins**

- 배치 처리에서 좋은 처리량을 위해 연산은 가능한 한 **로컬(한 머신)** 에서 이루어져야 함
- **해결책**: 사용자 DB의 복사본을 같은 분산 파일시스템에 넣어, HDFS에서 함께 처리
  - 운영 중인 사용자 DB에 배치 작업이 직접 대량 쿼리를 날리지 않도록, 사용자 DB의 스냅샷/복사본을 HDFS 같은 분산 파일시스템에 파일 형태로 적재한 뒤, 이벤트 로그와 사용자 데이터 양쪽을 MapReduce 배치 작업으로 조인한다.
  - 운영 DB에 직접 병렬 쿼리를 날리기X
  - 사용자 DB의 스냅샷을 HDFS에 분산 파일로 적재한 뒤, MapReduce의 sort/shuffle/reduce 메커니즘으로 join하라.
- Mapper의 역할: 각 입력 레코드에서 키와 값을 추출
  - **활동 이벤트 mapper**: user ID를 키로, 활동 이벤트를 값으로
  - **사용자 DB mapper**: user ID를 키로, 생년월일을 값으로
- MapReduce 프레임워크가 키로 파티셔닝하고 정렬하면, 같은 user ID의 활동 이벤트와 사용자 레코드가 reducer 입력에서 **인접**하게 됨
- **Secondary Sort**
  - MapReduce 작업이 레코드를 정렬할 때, reducer가 항상 **사용자 DB 레코드를 먼저** 보고, 그 뒤에 활동 이벤트가 타임스탬프 순서로 오도록 정렬할 수 있음
  - Reducer는 생년월일을 로컬 변수에 저장한 뒤, 같은 user ID의 활동 이벤트를 순회하며 결과를 출력
  - 특성: **네트워크 요청이 전혀 필요 없음**

### **Bringing related data together in the same place**

- Mapper가 키-값 쌍을 emit할 때, 키는 **배달 주소** 역할: 같은 키의 모든 쌍이 같은 목적지(reducer)로 전달됨
- MapReduce는 **물리적 네트워크 통신**(데이터를 올바른 머신에 전달)과 **애플리케이션 로직**(데이터 처리)을 분리
- 부분 실패(노드 크래시 등)도 MapReduce가  재시도하여 애플리케이션 로직에 영향 없음 → 부분 실패 걱정이 없음

### **GROUP BY**

- 조인 외에 "관련 데이터를 한 곳에 모으기"의 또 다른 일반적 사용: **키로 레코드 그룹핑** (SQL의 `GROUP BY`)
  - 예: 레코드 수 세기 (`COUNT(*)`), 특정 필드 값의 합 (`SUM(fieldname)`), 상위 k개 레코드 선택
- 그룹핑과 조인은 MapReduce 위에서 구현할 때 매우 유사하게 보임
- **Sessionization (세션화)**: 특정 사용자의 모든 활동 이벤트를 수집하여 행동의 시퀀스를 파악하는 과정
  - 예: A/B 테스팅, 마케팅 활동의 가치 판단

### **Handling skew (쏠림/핫스팟 처리)**

- "같은 키의 모든 레코드를 한 곳에 모으기" 패턴은 특정 키에 매우 많은 데이터가 집중될 때 문제
  - 예: 소셜 네트워크의 유명인은 **수백만 명의 팔로워** -> **linchpin objects** 또는 **hot keys**
- 하나의 reducer가 다른 것보다 훨씬 많은 레코드를 처리 -> **심각한 쏠림(skew, hot spots)**
  - MapReduce 작업은 **가장 느린 reducer**를 기다려야 하므로 전체 작업이 느려짐
- 쏠림 처리 알고리즘:
  1. **Pig의 skewed join**: 샘플링으로 핫 키 판단 -> 핫 키 레코드를 **여러 reducer에 무작위 배정**, 다른 쪽 입력의 해당 핫 키 레코드는 **모든** 해당 reducer에 복제
  2. **Crunch의 sharded join**: 핫 키를 **명시적으로 지정**
  3. **Hive의 skewed join 최적화**: 핫 키를 테이블 메타데이터에 명시하고, 핫 키에 대해서는 **map-side join** 사용
- 핫 키 그룹핑의 2단계 집계:
  1. 첫 번째 MapReduce: 핫 키 레코드를 **무작위로 여러 reducer에 분배**, 각 reducer가 부분 집계값 출력
  2. 두 번째 MapReduce: 모든 reducer의 부분 결과를 **하나의 키당 하나의 값으로 결합**

## **Map-Side Joins**

- Reduce-side join은 실제 조인 로직이 **reducer에서** 수행됨
  - Reduce-side join 장점: 입력 데이터에 대한 어떤 가정도 필요 없음
- 입력 데이터에 대해 **특정 가정을 할 수 있다면**, **map-side join**으로 더 빠르게 조인 가능
  - 특정 가정 예시:
    - 두 입력 데이터가 같은 key 기준으로 정렬되어 있다.
    - 두 입력 데이터가 같은 방식으로 파티셔닝되어 있다.
    - 작은 테이블 하나를 모든 mapper 메모리에 올릴 수 있다.
  - Map-side join 장점: **reducer 없이, 정렬 없이** 작동하는 축소된 MapReduce 작업
  - Map-side join 단점: 정렬, reducer로의 복사, 병합 과정이 **매우 비용이 큼**

### **Broadcast hash join**

- 큰 데이터셋과 작은 데이터셋을 조인할 때 가장 단순한 map-side join 방식
- 작은 데이터셋이 각 매퍼의 메모리에 완전히 적재될 수 있을 만큼 작아야 함
- 매퍼가 시작되면 작은 데이터를 읽어 **인메모리 해시 테이블로 구축**하고, 큰 입력을 스캔하면서 해시 테이블에서 조회
- "broadcast": 큰 입력의 각 파티션을 처리하는 매퍼가 작은 입력 전체를 읽기 때문
- Pig("replicated join"), Hive("MapJoin") 등에서 지원

### **Partitioned hash join**

- 두 입력이 **같은 방식으로 파티셔닝**되어 있으면, 각 파티션에 독립적으로 해시 조인 적용 가능
  - 각 매퍼가 더 작은 양의 데이터만 해시 테이블에 적재하면 되는 장점
  - 두 입력의 파티션 수, 키, 해시 함수가 동일해야 함
  - Hive에서는 **bucketed map join**이라고 부름

### **Map-side merge join**

- 입력 데이터셋이 같은 키로 **파티셔닝** + **정렬**까지 되어 있을 때 사용
- 입력이 메모리에 들어갈 만큼 작을 필요 없음
- 매퍼가 두 입력 파일을 키 오름차순으로 증분적으로 읽으며 같은 키의 레코드를 매칭

### **MapReduce workflows with map-side joins**

- map-side join vs reduce-side join의 선택은 다운스트림 작업에 영향
  - **reduce-side join 출력**: 조인 키로 파티셔닝 및 정렬됨
  - **map-side join 출력**: 입력과 동일한 방식으로 파티셔닝 및 정렬됨
- Hadoop 생태계에서는 데이터의 물리적 레이아웃 메타데이터를 **HCatalog**과 **Hive metastore**에서 관리

### **The Output of Batch Workflows**

- 배치 처리는 OLTP도 아니고 분석도 아님
  - 보고서가 아닌 **다른 종류의 구조**를 출력하는 경우가 많음

### **Building search indexes (검색 인덱스 구축)**

- Google의 원래 MapReduce 용도: 검색 엔진을 위한 인덱스 구축
- 전문 검색(full-text search) 인덱스는 term dictionary 파일로, 키워드를 조회하면 해당 키워드를 포함하는 문서 ID 목록(postings list)을 찾을 수 있음
- 고정된 문서 집합에 대해 인덱스를 구축하기에 배치 프로세스가 매우 효과적

### **Key-value stores as batch process output**

- 머신러닝 시스템 구축: 분류기(스팸 필터, 이상 탐지, 이미지 인식), 추천 시스템 등
- 배치 작업에서 직접 DB에 쓰는 것이 나쁜 이유:
  1. 레코드마다 네트워크 요청을 보내면 배치 작업의 정상 처리량보다 굉장히 많이 느림
  2. 모든 매퍼/리듀서가 동시에 같은 DB에 쓰면 DB가 overwhelm 당함
  3. MapReduce의 all-or-nothing 보장이 깨짐 -- 부분 완료된 작업의 결과가 외부에 노출될 위험
- **더 나은 해결책**: 배치 작업 내부에서 **새로운 데이터베이스를 직접 구축**하여 출력 디렉토리에 파일로 기록
  - 지원 도구: **Voldemort**, **Terrapin**, **ElephantDB**, **HBase bulk loading**
  - 서버가 기존 데이터 파일로 요청을 계속 서비스하면서, 새 데이터 파일 복사 완료 후 **원자적으로(atomically)** 전환
  - 문제 발생 시 기존의 불변 파일로 쉽게 롤백 가능

### **Philosophy of batch process outputs**

- **유닉스 철학**과 동일: 입력을 읽고 출력을 쓰되, 입력은 변경하지 않고, 이전 출력은 새 출력으로 완전히 대체, 부작용(side effects) 없음
- 불변 입출력의 장점:
  1. **롤백 용이성**: 버그가 있는 코드를 배포해도 이전 버전으로 롤백하고 재실행하면 됨 (**human fault tolerance**)
  2. **비가역성 최소화(minimizing irreversibility)**: 롤백이 쉬우므로 기능 개발 속도를 높일 수 있음
  3. **자동 재시도**: 입력이 불변이고 실패한 태스크의 출력이 폐기되므로 자동 재시도가 안전
  4. **동일 파일의 다목적 사용**: 같은 파일 집합을 다양한 작업의 입력으로 사용 가능
  5. **관심사의 분리(separation of concerns)**: 로직과 배선(입출력 디렉토리 설정)을 분리하여 코드 재사용 가능

### **Comparing Hadoop to Distributed Databases**

- Hadoop = 유닉스의 분산 버전 (HDFS = 파일시스템, MapReduce = sort 유틸리티)
- MapReduce 논문(2004) 발표 시 완전히 새로운 것이 아니었음
  - 조인, 병렬 처리 알고리즘은 이미 **MPP(Massively Parallel Processing)** 데이터베이스에서 10년 이상 구현되어 있었음
- **차이**:
  - MPP DB는 클러스터에서 분석 SQL 쿼리의 병렬 실행에 집중
  - MapReduce + 분산 파일시스템은 **임의의 프로그램을 실행할 수 있는 범용 운영체제**에 가까움

### **Diversity of storage**

- 데이터베이스: 특정 모델에 맞게 데이터를 구조화해야 함
- 분산 파일시스템: 단순한 바이트 시퀀스, 어떤 데이터 모델/인코딩이든 가능
- Hadoop은 데이터를 먼저 HDFS에 무차별적으로 덤프하고 나중에 처리 방법을 결정할 수 있게 함
  - **data lake** 또는 **enterprise data hub**라 부름
  - **schema-on-read** 접근법: 데이터를 읽을 때 스키마를 적용
  - **Sushi principle: "raw data is better"**
    - 날 것의 데이터가 있으면 여러번의 변경을 입히기 편함

### **Diversity of processing models**

- **MPP 데이터베이스**: 모놀리식, SQL에 최적화
  - 그러나 머신러닝, 추천 시스템, 전문 검색 인덱스, 이미지 분석 등은 **SQL로 표현할 수 없음**
- MapReduce는 엔지니어가 대규모 데이터셋에 대해 자신의 코드를 쉽게 실행할 수 있는 능력을 제공
- Hadoop 위에 다양한 처리 모델이 **하나의 공유 클러스터**에서 공존 가능
  - 예: HBase(OLTP DB)와 Impala(MPP 분석 DB)가 모두 HDFS를 저장소로 사용

### **Designing for frequent faults**

- **MPP 데이터베이스**: 노드 장애 시 전체 쿼리를 중단하고 재시작. 쿼리가 보통 짧으므로 재시도 비용이 크지 않음
- **MapReduce**: 개별 map/reduce 태스크 실패를 작업 전체에 영향 없이 재시도로 허용
  - 대규모 데이터를 오래 처리하는 작업에 더 적합
- MapReduce가 빈번한 태스크 종료를 허용하도록 설계된 이유:
  - Google은 **온라인 프로덕션 서비스와 오프라인 배치 작업이 같은 머신**에서 실행되는 혼합 사용 데이터센터
  - 높은 우선순위 태스크가 낮은 우선순위 태스크를 선점(preempt) 가능
  - MapReduce 작업은 저우선순위로 실행 -> **고우선순위 프로세스가 가져간 나머지 남은 자원을 효율적으로 사용하는 방식**
  - Google에서 1시간 MapReduce 태스크가 종료될 확률: 약 **5%** (하드웨어 장애보다 한 자릿수 높음)
  - 프로세스를 임의로 종료할 수 있는 자유가 컴퓨팅 클러스터의 자원 활용을 더 좋게 함

## **Beyond MapReduce**

- MapReduce는 분산 시스템을 위한 여러 프로그래밍 모델 중 **하나에 불과**
- MapReduce는 매우 **robust** 하여 비신뢰적인 멀티 테넌트 시스템에서도 대량의 데이터를 처리 가능하지만, 다른 도구들은 특정 처리에 있어 **수십 배 이상 빠를** 수 있다
- raw MapReduce API를 사용해 복잡한 처리 작업을 구현하는 것은 상당히 어렵고 수고스러움
  - **Pig, Hive, Cascading, Crunch** 같은 고수준 추상화가 만들어졌으나, MapReduce 실행 모델 자체의 문제는 해결되지 않음

### **Materialization of Intermediate State**

- 한 작업의 출력이 같은 팀의 다른 하나의 작업에만 사용되는 경우, 분산 파일시스템의 파일은 단순히 **중간 상태(intermediate state)** 에 불과
- 이 중간 상태를 파일로 기록하는 과정을 구체화(materialization)라고 함
- Unix 파이프는 중간 상태를 완전히 구체화하지 않고, 작은 인메모리 버퍼만 사용하여 출력을 점진적으로 **스트리밍**
- MapReduce의 중간 상태 완전 구체화의 단점:
  1. 선행 작업의 모든 태스크가 완료되어야만 후행 작업을 시작 가능 
        1. straggler task가 전체 워크플로를 느리게 만듦
  2. Mapper가 종종 불필요 
        1. reducer 출력이 mapper 출력과 같은 형태라면 직접 연결 가능
  3. 분산 파일시스템에 중간 상태를 저장하면 여러 노드에 복제됨
    1. 임시 데이터에 대해 과도한 처리

### **Dataflow engines**

- 이 문제를 해결하기 위해 새로운 엔진이 개발됨: **Spark, Tez, Flink**
- 전체 워크플로를 **하나의 작업으로 처리**하며, map과 reduce의 엄격한 교대를 따르지 않고 더 유연하게 조합 가능
  - 이 함수들을 operators라 부름
- 오퍼레이터 간 연결 옵션:
  - 키 기반 재파티셔닝 및 정렬 (shuffle과 동일)
  - 여러 입력을 같은 방식으로 파티션하되 정렬 생략
    - 파티션된 해시 조인에 유용
  - 브로드캐스트 해시 조인
    - 한 연산자의 출력을 모든 파티션에 전송
- MapReduce 대비 장점:
  1. 정렬 등 비용이 큰 작업은 실제로 필요한 곳에서만 수행
  2. 불필요한 map 태스크 제거
  3. 스케줄러가 locality(지역성) 최적화 가능 
        1. 데이터를 생산/소비하는 태스크를 같은 머신에 배치
  4. 중간 상태를 로컬 디스크나 메모리에 저장 ****
    1. HDFS 복제 불필요
  5. 연산자가 입력 준비되는 즉시 실행 시작 가능
  6. 기존 JVM 프로세스를 재사용 
        1. 태스크마다 새 JVM 시작 오버헤드 절감

### **Fault tolerance**

- **MapReduce**: 중간 상태를 분산 파일시스템에 완전 구체화하여 내구성 확보. 태스크 실패 시 다른 머신에서 재시작
- **Spark**: **RDD (Resilient Distributed Dataset)** 추상화로 데이터의 계보(ancestry)를 추적하여 재계산
- **Flink**: 연산자 상태를 **체크포인팅(checkpointing)** 하여 장애 발생 시 해당 연산자부터 재개
- 재계산 시 연산이 **deterministic(결정적)** 인지가 중요
  - 비결정적 행동의 원인: 해시 테이블 순회 순서, 난수, 시스템 시계, 외부 데이터 소스
  - 재계산이 항상 최선은 아님 → 중간 데이터가 소스보다 작거나 계산이 CPU 집약적이면, 파일로 구체화하는 것이 재계산보다 저렴할 수 있다
- MapReduce가 각 명령의 출력을 **임시 파일**에 쓰는 것이면 데이터플로 엔진는 **Unix 파이프**에 더 가까움
  - Flink은 특히 파이프라인 실행(pipelined execution) 아이디어를 중심으로 설계

### **Graphs and Iterative Processing**

- 배치 처리 맥락에서 그래프를 보면, 목표는 전체 그래프에 대한 **오프라인 처리나 분석**을 수행하는 것
  - 머신러닝의 추천 엔진, 랭킹 시스템 등에서 자주 발생
  - **PageRank**: 가장 유명한 그래프 분석 알고리즘 중 하나
- 많은 그래프 알고리즘은 한 번에 한 간선씩 순회하며, 어떤 조건이 충족될 때까지 반복
  - "완료될 때까지 반복"은 단일 패스만 하는 MapReduce로는 표현 불가 -> **반복적(iterative)** 스타일로 구현
  - MapReduce로 구현하면 매우 비효율적: 그래프의 작은 부분만 변경되어도 매 반복마다 전체 입력을 다시 읽어야 함

### The Pregel processing model

- 그래프 배치 처리의 최적화로 **BSP(Bulk Synchronous Parallel)** 계산 모델이 등장
  - ex: Apache Giraph, Spark의 GraphX API, Flink의 Gelly API
- 한 정점이 다른 정점에 **메시지를 보낼** 수 있으며, 보통 그래프의 간선을 따라 전송
- 각 반복(iteration)에서 각 정점에 대해 함수가 호출되며, 이전 반복에서 전송된 모든 메시지가 전달
  - MapReduce와의 차이: vertices**은 한 반복에서 다음 반복까지 자신의 상태를 메모리에 기억** -> 함수는 새로운 수신 메시지만 처리
- **액터 모델(actor model)과 유사하나, 전달을 보장한다는 차이가 있음**:
  - 정점 간 상태와 메시지가 **내결함성**을 갖고 **내구적(durable)**
  - 통신이 **고정된 라운드(fixed rounds)** 로 진행
  - 프레임워크가 모든 메시지 전달을 **보장** (액터는 이런 보장 없음)
- 내결함성: 반복 끝에 모든 정점의 상태를 **주기적으로 체크포인팅**하여 달성
  - 노드 장애 시 마지막 체크포인트로 **롤백하고 재시작**

### **Parallel execution**

- 정점은 정점 ID로만 메시지를 보내며, 프레임워크가 파티셔닝, 정점 배치, 메시지 라우팅을 담당
- 최적 파티셔닝을 찾는 것은 어려워 실제로는 임의 파티셔닝되는 경우가 많음
  - 결과적으로 **이종 머신간 통신 오버헤드**가 많음
- 그래프가 단일 컴퓨터 메모리에 들어간다면, 단일 머신 알고리즘이 분산 배치 처리보다 **더 빠를 가능성이 높다**
  - 메모리보다 크더라도 **GraphChi** 같은 프레임워크로 단일 머신 처리 가능
  - 단일 머신에 안 들어갈 정도로 큰 경우에만 분산 접근법이 불가피

### **High-Level APIs and Languages**

- 분산 배치 처리 실행 엔진이 성숙해짐에 따라 관심이 **프로그래밍 모델 개선, 처리 효율성 향상, 문제 범위 확장**으로 이동
- **Hive, Pig, Cascading, Crunch** 같은 고수준 언어/API가 인기를 얻음
- **Spark와 Flink**도 자체 고수준 데이터플로 API를 포함
  - **관계형 스타일의 빌딩 블록**: 필드 값 기반 조인, 키 기반 그룹화, 조건 필터링, 집계 등

### **The move toward declarative query languages**

- 조인을 관계형 연산자로 지정하면, 프레임워크가 최적 조인 알고리즘을 **자동으로 결정** 가능
  - **Hive, Spark, Flink**에는 비용 기반(cost-based) 쿼리 옵티마이저가 있음
- MapReduce는 **함수 콜백(function callbacks)** 아이디어를 기반으로 구축 -- 임의의 코드를 자유롭게 실행 가능
  - 이 자유가 MapReduce를 **MPP 데이터베이스와 구분해온 핵심 특성**
- 데이터플로 엔진도 선언적 기능을 도입:
  - 단순 연산이 선언적으로 표현되면, **컬럼 지향 스토리지 레이아웃**을 활용하여 필요한 컬럼만 읽을 수 있음
  - **벡터화 실행(vectorized execution)**: CPU 캐시에 친화적인 타이트한 내부 루프에서 함수 호출을 회피
  - Spark는 **JVM 바이트코드 생성**, Impala는 **LLVM으로 네이티브 코드 생성**
- 배치 처리 프레임워크가 점점 **MPP 데이터베이스와 유사**해지면서도, 임의 코드를 실행하는 **확장성의 유연성**은 유지

### **Specialization for different domains**

- 표준적 처리 패턴에 대한 공통 빌딩 블록의 **재사용 가능한 구현**이 가치 있음
- 점점 중요해지는 도메인: **통계적/수치적 알고리즘** (머신러닝 애플리케이션)
  - **Mahout**: MapReduce, Spark, Flink 위에 다양한 머신러닝 알고리즘 구현
  - **MADlib**: 관계형 MPP 데이터베이스(Apache HAWQ) 내부에 유사 기능 구현
- **k-nearest neighbors** 같은 공간 알고리즘, **근사 검색(approximate search)** 도 배치 처리에서 활용
- 배치 처리 시스템이 고수준 선언적 연산자를 갖추고, MPP 데이터베이스가 더 프로그래밍 가능해지면서 **둘은 점점 비슷해지고 있다**
